# MOE 3GEMM GPU Implementation Summary

## 1. Architecture Overview

### What is MoE 3GEMM?

Mixture of Experts (MoE) is an architecture where each transformer layer routes tokens to a subset of "expert" MLPs via a learned router. In Qwen3.5 MoE, each expert MLP computes:

```
y = down(up(x) * silu(gate(x)))
```

where gate, up, down are three separate GEMM operations — hence "3GEMM". The "3GEMM fused compressed" primitive runs all three GEMMs + SwiGLU activation + routing + reduce in a single GPU primitive, avoiding intermediate global memory round-trips.

### Model Parameters (Qwen3.5 MoE)

| Parameter | Value |
|-----------|-------|
| HIDDEN_SIZE | 2048 |

### Execution Pipeline (Decode Path — Single Token)

```
softmax_topk(router_logits) → expert_list + routing_weights
  ↓
mlp_gate_up(x, gate_w, up_w) → y = up(x) * silu(gate(x))   [per expert]
  ↓
mlp_down(y, down_w) → z = down(y) * routing_weight           [per expert]
  ↓
mlp_reduce(z) → final = Σ z_i                                [across experts]
```

Dispatch: `gws = {num_experts, SUBGROUP_SIZE, INTERMEDIATE_SIZE/N_BLOCK}`, `lws = {1, SUBGROUP_SIZE, SUBGROUP_NUM}`

Constants: `N_BLOCK=4`, `SUBGROUP_NUM=8`, `SUBGROUP_SIZE=32` (XE2) or `16` (older).

### Execution Pipeline (Prefill Path — Multiple Tokens)

Uses micro_gemm (oneDNN matmul) per expert instead of custom GEMV kernels:

```
softmax_topk → mask_gen → gather(tokens) → micro_gemm_up → micro_gemm_gate(+silu+binmul) → micro_gemm_down → scatter_reduce
```

## 4. Weight Quantization Support

### Asymmetric (u4/u8)
- `dequant = (weight - zero_point) * scale`
- Optimized via `xg_sum`: precompute `sum(x)` per group → `result = dot(x, w) * scale - xg_sum * zp * scale`
- ZP stored as packed u4 (2 values per byte for 4-bit) or u8

### Symmetric (i4/i8)
- `dequant = weight * scale` (no zero point)
- `HAS_ZP=0` → all ZP code compiled out via `#if HAS_ZP`
- `WEIGHT_IS_SIGNED=1` → sign-extension macros `DEQUANT_4BIT_LO/HI`, `DEQUANT_8BIT`
- `xg_sum` computation skipped entirely when `HAS_ZP=0` (optimization D)
- Dummy ZP tensor (all zeros, matching weight type) passed to satisfy kernel signature

### Key Dequantization Macros (in .cl file)
```c
// Unsigned (HAS_ZP=1, WEIGHT_IS_SIGNED=0):
DEQUANT_4BIT_LO(v) → convert_half((v) & 0x0F)
DEQUANT_4BIT_HI(v) → convert_half((v) >> 4)

// Signed (HAS_ZP=0, WEIGHT_IS_SIGNED=1):
DEQUANT_4BIT_LO(v) → convert_half((char)(((v) & 0x08) ? ((v) | 0xF0) : ((v) & 0x0F)))
DEQUANT_4BIT_HI(v) → convert_half((char)((((v)>>4) & 0x08) ? (((v)>>4) | 0xF0) : (((v)>>4) & 0x0F))))
```

## 5. Kernel Architecture (moe_3gemm_swiglu_mlp.cl)

### Preprocessing: Activation Loading to SLM

Both `mlp_gate_up` and `mlp_down` start by loading activation from global memory to SLM:

- **4-bit path** (`WEIGHT_COMPRESSEION_DT==0`): Interleaves even/odd elements for optimal sub-group block read alignment
- **8-bit/f16 path**: Direct copy to SLM
- **xg_sum**: Per-group activation sum, computed only when `HAS_ZP=1` (asymmetric quantization needs it for ZP compensation)

### GEMV Functions

| Function | Weight Type | Purpose |
|----------|-------------|---------|
| `gate_up_gemv_n2x_u4` | u4/i4 | **Active** — two-call gate+up GEMV (better perf) |
| `gate_up_gemv_n2x_u8` | u8/i8 | **Active** — two-call gate+up GEMV |
| `gate_up_gemv_n2x_f16` | f16 | **Active** — two-call gate+up GEMV |
| `gate_up_gemv_fused_u4` | u4/i4 | Experimental fused gate+up (slower, commented out) |
| `gate_up_gemv_fused_u8` | u8/i8 | Experimental fused gate+up (slower, commented out) |
| `gate_up_gemv_fused_f16` | f16 | Experimental fused gate+up (slower, commented out) |
| `down_gemv_n2x_u4/u8/f16` | various | Down projection GEMV × routing_weight |

### Entry Points (Kernels)
- `mlp_gate_up`: Loads activation → (optional) scalar gate for shared expert → fused gate+up GEMV
- `mlp_down`: Loads activation → down GEMV × routing_weight
- `mlp_reduce`: Cross-expert sum reduction

### Shared Expert Integration (Decode)
- Shared expert treated as expert `MAX_TOPK` (extra workgroup in dim-0)
- Scalar gate: workgroup-wide parallel dot-product reduction, sigmoid, writes `routing_weights[MAX_TOPK]`
- Same GEMV functions used for both sparse and shared experts

## 6. Optimizations Applied

### A. Fused Gate+Up GEMV — Attempted but Reverted

**Hypothesis**: Fusing gate and up into a single function would save 1 SLM read per group, eliminate `y[]` intermediate global write, and exploit ILP between gate/up weight loads.

**Result**: Fused version (`gate_up_gemv_fused_u4`) is **slower** than two separate `gate_up_gemv_n2x_u4` calls. Currently commented out in code.

**Root Cause Analysis** (3 factors, by impact):

1. **Register pressure → occupancy drop (primary)**
   - Non-fused per-call: ~10-12 GRF (a×4, b×2, sum×2, accum×2, scale/zp×2-4)
   - Fused: ~20-24 GRF (a×4, gb/gb2/ub/ub2×4, gsum/usum×4, gate_all/up_all×4, 2× scale/zp×4-8)
   - 2× register allocation → fewer concurrent HW threads → lower occupancy → exposed global memory latency (~300+ cycles)
   - For memory-bound kernel (~4 FLOPs/byte), occupancy is the critical factor for hiding memory latency

2. **Cross-weight memory access pattern → cache thrashing**
   - Non-fused: each call accesses one contiguous weight region (good L3 cache locality)
   - Fused: each iteration reads from 4 disjoint addresses (gate_B, gate_B+K/2, up_B, up_B+K/2) → 2× working set, cache line competition

3. **Instruction cache pressure**
   - Inner loop `K/FAKE_GROUP_SIZE = 16` iterations fully unrolled
   - Fused: ~384 instructions (16×24) vs non-fused: ~192 instructions (16×12)
   - 2× instruction footprint may cause I-cache misses on XE2

**Key Lesson**: For memory-bound GEMV, **low register pressure + high occupancy + sequential memory access** always beats reducing SLM/ALU operations. The saved SLM read (a few cycles) is negligible vs the occupancy loss (hundreds of cycles of exposed memory stall).

### B. Parallelized Shared Expert Scalar Gate

**Problem**: Single thread computing dot(x, gate_weight) over HIDDEN_SIZE=3584.

**Solution**: Workgroup-wide parallel reduction (256 threads, ~14 iterations each), two-level reduce (sub_group + cross-subgroup SLM).

### C. Symmetric Quantization xg_sum Skip (Optimization D)

**Problem**: `xg_sum` (activation group sum for ZP compensation) computed unconditionally, wasted cycles on symmetric quantization path.

**Solution**: Guard all `xg_sum` computation and SLM storage with `#if HAS_ZP`. Symmetric path (HAS_ZP=0) skips: per-element accumulation, sub_group_reduce_add(), SLM write — in both gate_up and down kernels.

### D. Shared Expert Transformation Fix

**Problem**: Pattern matcher collision — standard MOE matcher greedily consumed the inner MOE node before the shared expert matcher could match the full Add(MOE, SharedExpert) pattern.

**Solution**: Rejection predicate `!ov::is_type<ov::op::v1::Add>` on standard matcher; context-aware `replace_node` that swaps the correct root.

## 7. Roofline Analysis

INT4 GEMV on XE2 GPU:
- **Operational intensity**: ~4 FLOPs/byte (0.5B weight per element, 2 FLOPs per multiply-add)
- **Compute crossover**: ~100 FLOPs/byte on XE2
- **Regime**: Purely **memory-bound** (25× below crossover)
- **Implication**: Performance is dominated by memory bandwidth utilization, which depends on **occupancy** (to hide latency) and **cache efficiency** (to maximize effective bandwidth). Reducing SLM or ALU ops has negligible impact — any optimization that increases register pressure or disrupts memory access patterns will hurt performance even if it reduces total instruction count.

## 9. Known Limitations & Future Work

1. **No gate+up fusion for prefill path**: Prefill uses oneDNN micro_gemm (matrix-matrix multiply), not custom GEMV. Fusion only applies to decode (single-token GEMV).
2. **Shared expert prefill uses separate oneDNN matmuls**: Not integrated into micro_gemm dispatch; runs sequentially after sparse expert micro_gemms.
3. **Fixed INTERMEDIATE_SIZE assumption**: Shared expert assumed to have same INTERMEDIATE_SIZE as sparse experts. Models with different-sized shared experts would need code changes.
4. **OpenCL kernel redefinition constraint**: All inline functions must be macros (not `inline` functions) because the `.cl` file is compiled multiple times (once per sub-kernel) into a single OpenCL program — `inline` functions cause "redefinition" errors.
