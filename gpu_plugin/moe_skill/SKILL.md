## Workflow

1. **Read references**: Read `reference.md` for file map, `SUMMARY.md` for architecture/history, and `grouped_gemm.md` for prefill-path details
2. **Understand pipeline**: Decode uses custom GEMV kernels (3-stage); Prefill uses oneDNN micro_gemm or grouped GEMM
3. **Identify bottleneck**: Use roofline analysis to determine if memory-bound (decode) or compute-bound (prefill large batch)
4. **Implement & test**: Build with `make ov_gpu_unit_tests -j$(nproc)`, run `--gtest_filter="*moe*"`
5. **Document**: Update `SUMMARY.md` with results; keep SKILL.md as concise reference

### Rules
- Follow existing code style and conventions
- Don't create new commits or push code until optimization is verified
- Don't modify oneDNN code directly
- Include before-and-after performance metrics and roofline efficiency ratios in documentation

## Architecture Overview

### MoE 3GEMM Pipeline

Each MoE layer routes tokens to a subset of expert MLPs via a learned router. Each expert computes:

```
y = down(up(x) * silu(gate(x)))
```

Three GEMMs (gate, up, down) + SwiGLU activation + routing + reduce are fused into a single GPU primitive `MOE3GemmFusedCompressed`, avoiding intermediate global memory round-trips.

### Execution Paths

| Path | Condition | Implementation | Key Characteristic |
|------|-----------|----------------|-------------------|
| **Decode** | `token_num == 1` | Custom OpenCL GEMV kernels | Memory-bound (~4 FLOPs/byte) |
| **Prefill (micro_gemm)** | `token_num > 1`, small-medium batch | oneDNN micro_gemm per expert | Per-expert loop |
| **Prefill (grouped GEMM)** | `token_num > 1`, large batch, `MOE_USE_GROUPED_GEMM_PREFILL=1` | oneDNN grouped matmul | Single dispatch for all experts |

### Decode Pipeline (Single Token)

```
softmax_topk(router_logits) → expert_list + routing_weights
  ↓
mlp_gate_up(x, gate_w, up_w) → y = up(x) * silu(gate(x))   [per expert]
  ↓
mlp_down(y, down_w) → z = down(y) * routing_weight           [per expert]
  ↓
mlp_reduce(z) → final = Σ z_i                                [across experts]
```

- **Dispatch**: `gws={num_experts, SUBGROUP_SIZE, INTERMEDIATE_SIZE/N_BLOCK}`, `lws={1, SUBGROUP_SIZE, SUBGROUP_NUM}`
- **Constants**: `N_BLOCK=4`, `SUBGROUP_NUM=8`, `SUBGROUP_SIZE=32` (XE2) / `16` (older)

### Prefill Pipeline (Multiple Tokens)

**Micro_gemm path**:
```
softmax_topk → mask_gen → gather(tokens) → micro_gemm_up → micro_gemm_gate(+silu+binmul) → micro_gemm_down → scatter_reduce
```

**Grouped GEMM path**:
```
softmax_topk → CPU mask_gen → grouped_matmul_gate → grouped_matmul_up → swiglu → grouped_matmul_down → scatter_reduce
```

## Quantization Support

| Type | Weight | Formula | Key Behavior |
|------|--------|---------|-------------|
| **Asymmetric** | u4/u8 | `(w - zp) * scale` | Uses `xg_sum` optimization for ZP compensation |
| **Symmetric** | i4/i8 | `w * scale` | `HAS_ZP=0` compiles out all ZP code; needs dummy ZP tensor |

**Macros** (not inline functions — `.cl` file compiled 3× into single program):
- `DEQUANT_4BIT_LO/HI`, `DEQUANT_8BIT` — sign-extension controlled by `WEIGHT_IS_SIGNED`
- `ZP_ADJUST_*` — ZP compensation, guarded by `#if HAS_ZP`

### Group Size Support

| group_size | Status | Notes |
|-----------|--------|-------|
| 128, 256, 512, per-channel | ✅ Supported | `FAKE_GROUP_SIZE=128`, no perf impact |
| 64 | ⬜ Planned | Reduce `FAKE_GROUP_SIZE` to 64; ~same perf (memory-bound) |
| 32 (XE2+u4) | ❌ Not feasible | Sub-byte granularity; fall back to oneDNN |

## Shared Expert Integration

- **Decode**: Fused into main kernels as expert `MAX_TOPK` (extra workgroup in dim-0)
- **Scalar gate**: Workgroup-wide parallel dot-product reduction (256 threads), sigmoid
- **Prefill**: Separate oneDNN matmul calls after main MoE (not yet fused into grouped dispatch)
- **Transformation**: Rejection predicate `!is_type<Add>` on standard matcher prevents greedy pattern collision; context-aware `replace_node` swaps correct root

### Shared Expert Op Extension
- `MOE3GemmFusedCompressed` extended with optional inputs (indices 11-20) for shared expert weights/scales/zp
- Uncompressed shared experts (FP16) use dummy scales (1.0) and zero-points (0.0)

## Grouped GEMM Integration (Prefill)

### DNNL_ARG_HINT_MAX_GROUP_SIZE

Runtime execute argument for dispatch optimization — eliminates recompilation when routing changes:

| Approach | Recompilations | Dispatch quality |
|----------|---------------|-----------------|
| Old: `max_variable_dim` baked into pd | Every routing change | Optimal but costly |
| **New: runtime hint at execute** | **Zero** | **Optimal** |

**Usage**:
```cpp
auto hint_md = dnnl::memory::desc::host_scalar(dnnl::memory::data_type::s32);
dnnl::memory hint_mem(hint_md, static_cast<int32_t>(max_tokens_per_expert));
// Pass in all three GEMM execute args:
{DNNL_ARG_HINT_MAX_GROUP_SIZE, hint_mem}
```

**Primitive caching**: One compiled primitive set (gate/up/down) per distinct `total_tokens`. Zero additional compilations from routing variation.

### CPU vs GPU Mask Generation
CPU mask gen is preferred — the GPU `moe_mask_gen.cl` kernel has an inherently serial inner loop (128 threads each scanning `tokens × topk` entries linearly). CPU benefits from branch prediction and cache locality on sequential reads. The CPU→GPU sync cost is justified.

### Performance Characteristics

| Input Tokens | Grouped better? | Notes |
|-------------|----------------|-------|
| < 4096 | No | Flat or slightly worse; use per-expert loop |
| 8192 | **Yes (-11.7% prefill)** | Clear benefit from single dispatch |
| 32768 | Neutral | Decode dominated by attention/KV cache |

## Roofline Analysis

### Decode (Memory-Bound)

- **Operational intensity**: ~4 FLOPs/byte (INT4 GEMV)
- **XE2 compute crossover**: ~239 FLOPs/byte
- **Regime**: Purely **memory-bound** (57× below crossover)
- **Implication**: Only optimizations reducing data movement or improving access patterns help. Reducing ALU ops is irrelevant.

## Optimization History

### Applied (Verified)

| Optimization | Impact | Path |
|-------------|--------|------|
| **xg_sum skip for symmetric** | Eliminates wasted ZP computation | Decode |
| **Parallel shared expert scalar gate** | 256-thread parallel reduction vs single-thread | Decode |
| **Shared expert transformation fix** | Enables shared expert fusion via predicate isolation | Graph |
| **DNNL_ARG_HINT_MAX_GROUP_SIZE** | Zero recompilations, exact dispatch | Prefill |

### Attempted but Reverted

| Optimization | Result | Root Cause |
|-------------|--------|-----------|
| **Fused gate+up GEMV** | Slower | 2× register pressure → occupancy drop; cache thrashing across disjoint weight regions; I-cache pressure from unrolled code. For memory-bound GEMV, occupancy > ALU savings. |

### Rejected (Analysis Showed Not Viable)

| Optimization | Reason |
|-------------|--------|
| GPU mask gen for grouped GEMM | GPU single-thread execution slower than CPU |
| Interleave gate+up weights (decode only) | Prefill+decode share weights; no separate layout possible |
| K-split across subgroups | Doesn't reduce memory traffic; adds barrier overhead |
