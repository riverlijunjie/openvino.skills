# `sdpa_micro__generate` standalone test

Standalone perf + correctness harness for the OpenVINO Intel-GPU **`micro_sdpa`
generate (decode)** kernel, extracted so it can be built and run **outside**
OpenVINO with only the oneDNN/gemmstone microkernel JIT and an OpenCL runtime.

It reproduces exactly what
`SDPAMicroGenerator(/*prefill=*/false, /*gqa_single_token=*/false)` does in
`src/plugins/intel_gpu/src/graph/impls/ocl_v2/` for a regular (non-paged) SDPA
decode step with an **int8, asymmetric, per-token compressed KV cache**.

---

## What it does

1. **Config selection** â€” picks the pre-tuned `sdpa_config_t` (unroll / work-group
   tiling) for the detected GPU architecture, mirroring
   `choose_config_xe2 / choose_config_xehpc / choose_config_xe3p` in the plugin.
2. **Microkernel generation** â€” builds the two oneDNN/gemmstone micro-GEMMs
   (`K^TÂ·Q` and `VÂ·S`) via `gemmstone::microkernel::selectGEMM`. This is **pure
   host code**: ngen emits GPU ISA into memory, so it runs on any machine
   regardless of GPU.
3. **Source assembly** â€” emits the OpenCL-C shims with `generateShim` and
   assembles the full kernel source the same way `kernels_cache.cpp` does:

   ```
   shim_kq + shim_vs                              (the "jit" section, prepended)
   + decoration macros + all JIT #defines
   + sdpa_micro.cl body                           (the "str" section)
   ```

   with the batch-header includes (`generic_vector_ops.cl`, `sdpa_utils.cl`,
   `tile_ops.cl`) inlined.
4. **Build + fuse** â€” `clBuildProgram`, then `gemmstone::microkernel::fuse()`
   patches the microkernel machine code into the program binary, then rebuilds
   from the patched binary (`clCreateProgramWithBinary`).
5. **Run + verify + benchmark** â€” fills Q/K/V (K/V quantized to int8 per-token),
   runs the kernel, compares against an f32 reference (relative-L2 < 2e-2), and
   times `--iters` iterations (ms/iter, GFLOP/s, KV GB/s).

## Build

```bash
./build_test.sh              # -O2 release  -> sdpa_micro_generate_test
DEBUG=1 ./build_test.sh      # -O0 -g       -> sdpa_micro_generate_test_dbg
```

The script derives all paths from the repo root and the
`build-x86_64-release` build directory. It requires that the Intel-GPU oneDNN
component has already been built in that directory (it links
`build-x86_64-release/.../onednn_gpu_install/lib/libopenvino_onednn_gpu.a` and
the per-arch generator objects).

### Stale-library note (important)

A prebuilt `libopenvino_onednn_gpu.a` on a dev box can be **stale** relative to
the current gemmstone headers â€” the `GEMMProblem` struct layout drifts between
checkouts, which silently corrupts `selectGEMM`/`transpose` at runtime
(segfault). To stay robust, `build_test.sh` re-packs the freshly compiled
`dnnl_gpu_intel_gemm_jit` + `generatorXE2/XE3/XE3P` object files into
`libgemmjit_fresh.a` and lists it **first** inside the link group
(`--start-group â€¦ --end-group` with `--allow-multiple-definition`), so the
up-to-date symbols win. Disable with `USE_FRESH_GEMM_JIT=0` on a known-consistent
build. If you change gemmstone sources, rebuild the objects first:

```bash
make -C build-x86_64-release/src/plugins/intel_gpu/thirdparty/onednn_gpu_build \
     -j"$(nproc)" dnnl_gpu_intel_gemm_jit generatorXE2 generatorXE3 generatorXE3P
```

## Run

```bash
./sdpa_micro_generate_test [options]
```

| Flag | Meaning | Default |
|------|---------|---------|
| `--tokens N` | query length `Lq` (tokens generated this step) | 1 |
| `--seqs N` | batch / sequence count | 1 |
| `--history N` | past KV length; `Lk = history + tokens` | 4096 |
| `--head-dim N` | head size `D` (specialized for â‰¤ 128) | 128 |
| `--kv-heads N` | number of K/V heads (GQA groups) | 8 |
| `--heads N` | number of Q heads (â‰Ą kv-heads, multiple of it) | 8 |
| `--iters N` | timed iterations | 50 |
| `--causal 0\|1` | causal masking | 0 |
| `--kernel-dir P` | dir holding `sdpa_micro.cl` and `include/` | auto |
| `--dump-source` | write assembled OpenCL to `sdpa_micro_full.cl` | off |
| `--force-arch A` | override detected arch for the host JIT: `xe2\|xe3\|xe3p` | off |
| `--gen-only` | stop after host-side shim generation; do not touch the GPU | off |

The KV cache is **always int8, asymmetric, per-token** compressed (matching the
plugin path under test); scales and zero-points are `half`.

### Default scenario

`--tokens 1 --seqs 1 --history 4096 --head-dim 128 --kv-heads 8 --heads 8`
â†’ 1 query token attending over 4097 keys, 8 KV heads / 8 Q heads (no GQA
broadcast), `D = 128`, int8 KV.

## Hardware requirements

- **Steps 1â€“3 (host microkernel generation)** run on **any** machine â€” no GPU
  needed. Use `--gen-only` to validate them locally.
- **Steps 4â€“5 (GPU build + run)** require an Intel GPU with **XMX / systolic
  support**: **BMG (Xe2)** or **PTL (Xe3)**. On a gen9/UHD dev box the kernel
  build correctly fails ("No matching kernel" from `selectGEMM`, or
  `clBuildProgram` error) because there is no systolic int8 SDPA microkernel for
  gen9. Run the full test on the remote target â€” see
  [remote_machine.md](remote_machine.md).

### Validating the host path on a non-XMX box

```bash
# Generate BMG (Xe2) microkernels and dump the assembled source, no GPU touched:
./sdpa_micro_generate_test --gen-only --force-arch xe2 --dump-source

# Same for PTL (Xe3):
./sdpa_micro_generate_test --gen-only --force-arch xe3
```

Expected output (BMG/PTL, default scenario):

```
host JIT OK: KQ binary=79456B grfMin=81 barriers=0 systolic=1 \
             wg_tile_m=256 wg_tile_n=16 sg_per_wg_m=16 sg_per_wg_n=1
host JIT OK: VS binary=36576B grfMin=72 barriers=0 systolic=1
```

`--force-arch xe3p` depends on the catalog being built with Xe3p SDPA configs and
may report "No matching kernel" in this checkout. `xe_hpc` (PVC) is not offered
because its gmdid encoding differs and the catalog here targets BMG/PTL.

## How the configuration was extracted

- **Source kernel**: `src/plugins/intel_gpu/src/graph/impls/ocl_v2/sdpa_micro.cl`
  (entry `micro_sdpa`).
- **Host generator**:
  `src/plugins/intel_gpu/.../sdpa_gen_micro.cpp` (`init_microkernels`,
  `get_jit_constants`, `get_dispatch_data_func`, `get_arguments_desc`,
  `get_build_options`).
- The JIT `#define`s, GWS/LWS dispatch, argument order, and per-token int8
  quantization layout in this test are transcribed 1:1 from those functions for
  the non-paged, int8-asymmetric generate path.
- The `KERNEL()` / `OPTIONAL_SHAPE_INFO_ARG` / `FUNC()` macros normally injected
  by `kernel_generator.cpp` are supplied manually in the assembled source.

GWS/LWS (non-PA generate):

```
LWS = { sub_group_size, sg_per_wg, 1 }
GWS = LWS, then
      GWS[0] *= ceil_div(Lq, wg_tile_q)   // wg_tile_q = KQ "wg_tile_n"
      GWS[1] *= Hq                          // Q heads
      GWS[2] *= batch
scalars: d = v_head_size, k = Lk, q = Lq
```
