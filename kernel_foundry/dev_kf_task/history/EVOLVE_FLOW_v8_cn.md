## 1. 实验配置与目标

### 1.1 实验目标

v8 实验的核心目标是验证以下假设：**通过在 USER_INSTRUCTIONS 中提供显式的 DPAS 类型约束模板，可以从根本上解决之前实验中反复出现的类型错误问题，从而释放进化算法的真正潜力。**

具体而言，v8 的设计改进针对 v7 的三大失败根因：
1. temp=0.0 导致多样性完全丧失
2. feedback_llm 引入的提示噪声干扰主模型
3. 缺乏明确的 DPAS 操作数类型指导

### 1.2 v8 实验配置

| 配置项 | v8 值 | v7 值（对照） | 变更说明 |
|--------|--------|---------------|----------|
| `model` | claude-4-6-opus | claude-4-6-opus | 相同 |
| `temperature` | **0.1** | 0.0 | **关键变更：恢复最小随机性** |
| `max_iters` | 8 | 10 | 略减 |
| `branches_per_iteration` | 4 | 4 | 相同 |
| 总评估次数 | 32 | 40 | 减少 20% |
| `evolve_mode` | true | true | 相同 |
| `use_gradient_tracking` | true | true | 相同 |
| `gradient_sampling_weight` | 0.3 | — | 新增 |
| `use_optimization_aware_prompting` | true | true | 相同 |
| `exploration_strategy` | mutate | — | 明确指定 |
| `use_exploration_prompts` | true | true | 相同 |
| `include_inspirations` | true | true | 相同 |
| `include_hardware_specs` | true | — | 新增 |
| `feedback_llm` | **DISABLED** | gpt-5.3-codex | **关键变更：移除反馈模型** |
| `exploration/exploitation ratios` | **DISABLED** | 0.4/0.5 | **关键变更：移除数据库约束** |
| **USER_INSTRUCTIONS** | **含显式 DPAS 类型模板** | 无 | **最关键变更** |

### 1.3 关键新特性：显式 DPAS 类型约束

v8 实验在 USER_INSTRUCTIONS 中新增了以下强制性类型模板：

```
CRITICAL: For intel_sub_group_f16_f16_matrix_mad_k16, you MUST use:
    - First operand: short8 (NOT float8)
    - Second operand: int8 (NOT float8)
    - Accumulator: float8 (this one is float8)
```

这一约束直接解决了 v5/v7 实验中 100% 的 kernel 使用错误类型 `float8` 作为 A/B 操作数的致命问题。Intel DPAS 指令要求 A 矩阵以 `short8`（packed FP16）传入，B 矩阵以 `int8`（VNNI layout packed）传入，仅累加器使用 `float8`。

### 1.4 设计哲学

v8 的设计体现了"减法优化"思维：
- **减少**配置复杂度（移除 feedback_llm、database ratios）
- **增加**关键领域知识（显式类型模板）
- **恢复**必要随机性（temp=0.1）

核心洞察：**与其增加系统复杂度来"引导"模型，不如直接告诉模型关键约束条件。**

### 3.4 最佳 Kernel 架构详解 (Trial 7, 30.9ms)

**工作组配置**：
```c
__attribute__((reqd_work_group_size(16, 8, 1)))
// 16 lanes × 8 subgroups = 128 threads per workgroup
```

**Tile 分解**：
- 工作组处理 64M × 64N 的输出块
- 每个子组处理 8M × 64N 的子块
- TILE_K = 32，分两步执行 k16 DPAS

**内存层次**：
```
Global Memory (A, B)
    ↓ 协作加载 (128 threads)
Shared Local Memory (SLM)
    - A: 64×32 tiles (每线程加载 16 个 half 元素)
    - B: 32×64 tiles (VNNI format, 每线程加载 8 个 uint)
    ↓ sub_group_block_read
Registers
    ↓ DPAS
Output Registers → Global Memory (C)
```

**DPAS 调用模式**：
```c
// 正确的类型使用
short8 a_data = ...; // A 矩阵操作数 (packed FP16)
int8   b_data = ...; // B 矩阵操作数 (VNNI layout)
float8 acc = (float8)(0.0f); // 累加器

acc = intel_sub_group_f16_f16_matrix_mad_k16(a_data, b_data, acc);
```

**关键优化技巧**：
1. **VNNI 格式 B 矩阵**：B 以 `uint` 打包加载到 SLM，自然满足 DPAS 的 VNNI 布局要求
2. **边界安全分支**：使用 `m_safe`/`n_safe` 标志区分内部 tile 和边界 tile，内部 tile 跳过所有边界检查
3. **高效协作加载**：128 线程均匀分配加载任务，A 每线程 16 元素，B 每线程 8 元素
4. **向量化存储**：使用 `convert_half8(acc)` 直接向量写回

## 5. 关键技术发现

### 5.1 发现一：显式类型模板是最关键的单一变更

**证据链**：
- v5（无类型模板，多模型）：100% 使用错误的 `float8` 类型
- v7（无类型模板，opus temp=0）：100% 使用错误的 `float8` 类型
- v8（有类型模板，opus temp=0.1）：**100% 使用正确的 `short8`/`int8` 类型**

**结论**：对于 Intel DPAS 这种文档稀缺、类型反直觉的专用指令，模型无法仅凭通用知识推断正确类型。显式约束是必要且充分的解决方案。

### 5.2 发现二：temp=0.1 是进化模式的最低有效温度

**对比**：
- temp=0.0（v7）：40 次评估产生完全相同的代码，零多样性
- temp=0.1（v8）：32 次评估产生 8 次 elite 替换，架构从 1-SG 进化到 8-SG

**机制解释**：temp=0.1 提供的随机性刚好足够让模型在等价优化方案之间进行选择（如选择不同 tile 大小、不同子组数量），而不会引入影响代码正确性的噪声。

### 5.3 发现三：减少系统复杂度优于增加系统复杂度

v7 → v8 的三项"减法"操作（移除 feedback_llm、移除 database ratios、简化提示）全部带来正向效果：

| 移除项 | 原本效果 | 移除后效果 |
|--------|----------|-----------|
| feedback_llm | 添加额外提示上下文，可能干扰主模型决策 | 主模型决策更纯净 |
| database ratios | 强制固定探索/开发比例 | 让算法自然选择最优平衡 |
| 复杂配置 | 增加认知负载 | 关注核心：类型正确+适度随机 |

### 5.4 发现四：进化可发现架构级优化

v8 进化过程中发现的非平凡架构改进序列：

1. **1 子组 → 8 子组**：并行度提升 8x
2. **逐元素加载 → 协作块加载**：内存带宽利用率大幅提升
3. **无边界优化 → m_safe/n_safe 分支**：消除内部 tile 的边界检查开销
4. **直接 FP16 存储 → VNNI 预打包**：消除 DPAS 指令前的格式转换

这些优化不是简单的参数调优，而是结构性的架构重组，证明 LLM 引导的进化确实能发现有意义的优化路径。

### 5.5 发现五：v8 是首个超越参考基线的进化实验

| 实验 | 相对参考（33.9ms） | 性质 |
|------|-------------------|------|
| v5 | 1.0x（等于参考） | 失败 |
| v6 | 0.94x（慢于参考） | 部分成功 |
| v7 | 1.0x（等于参考） | 失败 |
| **v8** | **1.097x（快于参考）** | **首次突破** |

30.9ms < 33.9ms，加速比 1.097x。这证明在正确配置下，进化模式可以超越单次最佳生成的结果。

## 6. 优化建议

### 6.1 短期改进（下一次实验 v9）

| 优先级 | 建议 | 预期效果 | 风险 |
|--------|------|----------|------|
| **P0** | 使用 v4 的 11.4ms kernel 作为进化初始种子 | 从已知最优解开始进化，避免重复发现已知优化 | 低：最差情况不退化 |
| **P0** | 保留显式 DPAS 类型模板 | 维持 100% 类型正确率 | 无 |
| P1 | 提升温度至 0.15-0.2 | 在 v4 基础上探索更多变体 | 中：可能引入错误 |
| P1 | 增加迭代数至 16-20 | 给进化更多时间收敛 | 低：仅增加计算成本 |
| P2 | 在 USER_INSTRUCTIONS 中添加显式优化提示 | 引导模型发现特定优化 | 中：可能限制探索空间 |

### 6.2 中期改进

#### 6.2.1 温启动策略

```python
# 建议实现方式
initial_seed = load_kernel("v4_best_11.4ms.cl")
population = [initial_seed]
# 第一轮 mutate 直接在 11.4ms 基础上变异
```

从 11.4ms 而非 332ms 起步，可以跳过 v8 前 5 代的"重新发现"阶段，将全部计算预算用于超越 11.4ms。

#### 6.2.2 优化提示注入

在 USER_INSTRUCTIONS 中补充以下经验性优化提示：

```
Performance optimization techniques to consider:
1. Double-buffering: Overlap SLM loads with DPAS computation
2. Prefetching: Use async_work_group_copy for next K-tile
3. Register blocking: Maximize DPAS utilization per register load
4. Loop unrolling: Fully unroll K-loop if TILE_K <= 64
5. SLM bank conflict avoidance: Pad SLM arrays by +1 column
```

#### 6.2.3 自适应温度策略

```python
# 进化停滞时自动升温
if no_improvement_for(3_iterations):
    temperature = min(temperature * 1.5, 0.4)
# 发现改进时降温保持
if new_elite_found:
    temperature = max(temperature * 0.8, 0.1)
```

### 6.3 长期方向

1. **多目标进化**：不仅优化运行时间，同时优化 SLM 使用量和寄存器压力，寻找 Pareto 前沿
2. **跨 cell 知识迁移**：将一个 cell 中发现的优化模式（如 VNNI 预打包）自动注入相邻 cell 的提示
3. **自动超参数搜索**：对 TILE_M/TILE_N/TILE_K/NUM_SG 进行网格搜索，为每种组合运行独立进化
4. **混合策略**：前 N 代使用 temp=0.0 直接生成最优候选（如 v4 模式），然后切换到进化模式在最优解周围搜索

### 6.4 预期目标

| 时间框架 | 目标运行时间 | 加速比 | 依赖条件 |
|----------|-------------|--------|----------|
| v9（短期） | < 11.4ms | > 2.98x | v4 种子 + 16 迭代 |
| v10（中期） | < 8ms | > 4.2x | 双缓冲 + 预取 + 温启动 |
| 终极目标 | < 5ms | > 6.8x | 理论峰值性能的 ~70% |
