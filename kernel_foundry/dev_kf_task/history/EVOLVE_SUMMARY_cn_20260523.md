# KernelFoundry 进化式Kernel优化：综合实验报告

## 摘要

本报告总结了在 Intel Battlemage B580 GPU 上进行的 8 次（v4-v11）LLM驱动的进化式 kernel 优化实验。实验任务为 FP16 矩阵乘法 C[2048,2560] = A[2048,2048] × B[2048,2560]，利用 Intel XMX DPAS 指令。通过系统性地调整温度、种子 kernel、提示策略等参数，我们从 33.9ms 的参考基线优化到 **1.07ms**（31.7x加速），实现了 **20.9% 的 XMX 硬件利用率**。实验揭示了 LLM 进化式优化的核心机制、关键控制变量和微架构约束，为后续工作提供了明确方向。

---

## 1. 背景与原理

### 1.1 进化式Kernel优化的基本思想

传统 GPU kernel 优化依赖人类专家的手工调优，需要深厚的硬件知识和大量试错。KernelFoundry 提出了一种新范式：**以 LLM 作为"变异算子"，以进化算法框架（MAP-Elites）作为搜索策略**，自动发现高性能 GPU kernel。

核心流程：
```
初始种子 kernel → LLM 生成变异体 → 硬件评估（编译+正确性+性能）
         ↑                                          ↓
         ← MAP-Elites 档案选择下一代父代 ←──────────┘
```

### 1.2 MAP-Elites 质量多样性算法

不同于传统进化算法仅保留"最优个体"，MAP-Elites 维护一个**行为特征空间**中的精英档案：

- **4D行为空间**: memory_opt × compute_opt × parallelism_opt × esimd_opt，每维4级（0-3）
- **256个细胞**: 4^4 = 256 种可能的行为利基
- **精英选择**: 每个细胞保留该行为组合中的最优kernel
- **岛屿模型**: 4个子种群并行进化，通过迁移共享优秀个体

行为特征通过代码模式分析自动提取，例如：
- memory_opt: SLM使用、prefetch、bank conflict avoidance
- compute_opt: DPAS指令、loop unrolling、register blocking
- parallelism_opt: WG size、subgroup数量、协作加载

### 1.3 LLM 作为变异算子

与传统进化算法使用随机位翻转不同，LLM变异是**语义感知**的：

- **输入**: 当前最优kernel + 性能反馈 + 硬件规格 + 优化提示
- **输出**: 经过"思考"的改进版本（包含分析推理 + 新代码）
- **温度控制**: 决定变异幅度——低温（渐进微调）vs 高温（架构级重设计）

### 1.4 适应度函数

```
combined_score = perf_score + 3 × I(correct ∧ speedup > 0) × runtime_improvement

其中:
- perf_score: 性能分级 (1-5)
- runtime_improvement: 相对参考基线的加速比
- I(·): 正确性指示函数，仅正确且有加速的kernel得到fitness奖励
```

### 1.5 目标问题

```
硬件: Intel Battlemage G21 (B580)
  - 20 Xe2 cores
  - FP16 XMX 理论峰值: 96 TFLOPS (20 × 2048 ops/cycle × 2.4 GHz / 1024)
  - 内存带宽: ~456 GB/s
  - L2 Cache: ~24 MB

问题: FP16 GEMM
  - C[2048, 2560] = A[2048, 2048] × B[2048, 2560]
  - 计算量: 21.5 GFLOP
  - 理论最小时间: 0.224 ms (计算密集型, OI=731 FLOP/byte)
  
关键指令: intel_sub_group_f16_f16_matrix_mad_k16(short8 a, int8 b, float8 acc)
  - 每次调用: 8×16×16 = 2048 FP16 MADs = 4096 FLOPs
  - Subgroup size = 16
```

## 4. 关键发现

### 4.1 发现一：温度是进化质量的核心控制变量

```
温度效应图:

性能改善幅度
     ^
  31x|                    * v10 (0.25)
     |
     |
     |
     |
     |
  1.4x|              * v9 (0.2)
  1.1x|        * v8 (0.1)
  1.0x|──*─────*─────────────────────* ← v5(0.3-0.6), v7(0.0)
  0.9x|  v7   v6(0.1)               v11(0.3)
     +────+────+────+────+────+────+→ Temperature
          0   0.1  0.2  0.25  0.3

最优温度窗口: 0.25（足够产生架构突变，又不至于破坏已验证的结构）
```

**机制解释**：
- **T=0.0**: 解码确定性最高→每次生成几乎相同→零进化
- **T=0.1**: 微小随机性→仅能做同架构微调（重命名变量、调整常量）
- **T=0.2**: 中等随机性→可以改变tile尺寸、循环结构，但无法跨越架构边界
- **T=0.25**: 关键阈值→足以让模型"遗忘"某些固定模式（如B→SLM），产生全新架构
- **T=0.3**: 过高→在已优良的架构上做过多改变→退步

### 4.2 发现二：种子Kernel是突破性能的关键前提

| 种子状态 | 最佳进化结果 | 分析 |
|----------|-------------|------|
| 空（v5-v9）| 23.8ms | LLM需从头设计DPAS kernel，大量进化预算浪费在基础正确性上 |
| v4 best (v10) | 1.07ms | 种子已验证DPAS正确用法，进化预算100%用于性能优化 |
| v10 best (v11) | 1.31ms | 种子已经接近局部最优，高温导致偏离最优→退步 |

**核心洞察**: 种子的最佳选择是"正确但远未最优"的kernel——
- 太差（空）: 进化浪费在正确性探索上
- 太好（v10→v11）: 高温变异倾向于破坏已有优化
- 恰好（v4→v10）: 正确性已解决，性能有大量提升空间

### 4.3 发现三：架构级相变现象

v10 展现了经典的**相变**（phase transition）行为：

```
v10 进化轨迹:
  7.0 → 5.74 → 5.93 → 3.87 → 5.65 → 5.22 → [1.08] → 1.09 → ...
                                               ↑
                                        架构相变点 (Trial 6)
                                        3.87ms → 1.08ms (3.6x跳跃)
```

**相变前**: 所有kernel都使用 "A+B都放SLM" 的传统架构（渐进优化 3-7ms）
**相变后**: 发现 "A放SLM, B留global" 的非直觉架构（突破到1ms级别）

这说明性能景观中存在**适应度山谷**——必须跨越一个"反直觉"的中间态才能到达全局最优。温度0.25恰好提供了跨越这个山谷所需的"跳跃力"。

### 4.4 发现四：GPU微架构约束 > 算法级优化

v11证明了在B580上：

| 理论更优 | 实际更差 | 原因 |
|----------|----------|------|
| 双缓冲（overlap load/compute） | 单缓冲更快 | SLM占用↑ → 占用率↓ → 延迟隐藏差 |
| 128 WIs（更多并行） | 64 WIs更快 | 寄存器压力↑ → 溢出 |
| B放SLM（共享，减少带宽） | B从L2读更快 | L2已经够用，省下的SLM换来更多WG并发 |
| TILE_K=64（更多DPAS/barrier） | TILE_K=32更快 | SLM爆炸 + 加载循环过长 |

**核心教训**: 在GPU优化中，**占用率**（occupancy）和**延迟隐藏**（latency hiding）的影响往往大于**算法级优化**（双缓冲、更大tile）。2.2KB的SLM允许多个WG并发，比25KB的"完美"双缓冲方案更有效。

### 4.5 发现五：LLM模型选择至关重要

| 模型配置 | 结果 | 分析 |
|----------|------|------|
| sonnet + codex 混合 (v5) | 33.9ms 无进展 | 模型间风格不一致，无法积累进化 |
| opus 单独 (v4, v6-v11) | 最终1.07ms | 风格一致，推理深度足够 |

v5 使用了 claude-4-5-sonnet + gpt-5.3-codex + claude-4-6-opus 的混合配置，但 20 轮迭代毫无进展。原因：
- 不同模型的代码风格差异导致"子代"无法在"父代"基础上有效进化
- 较弱的模型（codex）可能破坏较强模型（opus）建立的优化结构

## 5. B580 最优Kernel架构解析

### 5.1 v10最优kernel（1.07ms）核心设计

```c
// 硬件映射:
//   WG tile: 32 rows × 64 cols
//   4 subgroups × 16 WIs = 64 WIs per WG
//   每个subgroup: 4个垂直8×16 DPAS = 32 rows × 16 cols
//   A → SLM (共享, 仅~2.2KB), B → global/L2 (直接读取)
//   K循环: TILE_K=32, 每K-tile 2次k16 DPAS步

关键优化:
1. 位操作索引 (>> 5, & 31) 替代除法/取模
2. 快速路径: tile完全在边界内时跳过边界检查
3. SLM padding (+2) 避免bank conflict
4. 协作加载: 64 WIs 每人16个元素填充32×32 A tile
```

### 5.2 为什么这个架构在B580上最优

```
性能分解 (估算):

  XMX计算: 8 DPAS × 4096 FLOP = 32,768 FLOPs per K-tile
  SLM加载: 16 half reads per WI × 64 WIs (A) ≈ 4 cycles
  B加载: 16 half reads per SG from L2 ≈ 8 cycles
  Barrier: ~10 cycles × 2 per K-tile
  
  关键: 小SLM (2.2KB) → 理论上可驻留 ~10 WGs per Xe-core
       → 当一个WG在等待barrier时,其他WG可以执行
       → 延迟隐藏是真正的加速来源
```

## 6. 进化式优化的优势

### 6.1 自动发现非直觉优化

- "A放SLM, B从global读" 不是任何教科书或手工优化指南推荐的策略
- 这是温度0.25的随机采样在第24次评估"意外"发现的
- 人类专家可能永远不会尝试这种"不对称"策略

### 6.2 高效搜索

- v10: 仅48次评估（14分钟）就找到31.7x加速
- 相当于每分钟产生 2.3x 的性能改善
- 对比人类手工调优可能需要数天到数周

### 6.3 100%正确性保证

- v10 所有48次评估全部产生正确结果
- 进化框架的验证机制确保只有正确kernel进入精英档案
- 消除了手工优化中"改快了但改错了"的风险

### 6.4 知识积累效应

- 每一代的分析和代码都作为下一代的上下文
- LLM的"分析"部分展示了对前代结果的理解和推理
- 这种"思考→实验→反馈"循环模拟了人类工程师的学习过程

## 7. 存在的问题与限制

### 7.1 对LLM推理能力的依赖

- 当前方案严重依赖 claude-4-6-opus 的代码推理能力
- 较弱的模型（v5的sonnet/codex混合）完全无法进化
- 推理成本高：每次生成需要 30-80 秒

### 7.2 无法突破 ~21% 利用率天花板

- v10达到20.9%后，v11的各种"更优"策略均无法突破
- 剩余79%的性能gap可能需要：
  - 编译器层面的优化（IGC指令调度）
  - 手工汇编（inline ISA）
  - 硬件特性的深层利用（如cooperative kernel launch）

### 7.3 高温变异的不可控性

- T=0.25 在v10产生突破，T=0.3 在v11导致退步
- 最优温度高度依赖于当前种子质量：
  - 远离最优的种子：需要高温（0.25-0.3）做大跨度探索
  - 接近最优的种子：需要低温（0.1-0.2）做精细调优
- 目前没有自动调温机制

### 7.4 种子选择的两难

- 空种子：浪费进化预算在基础正确性上
- 最优种子+高温：倾向于破坏已有优化
- 需要"恰到好处"的种子——正确但非最优

### 7.5 评估成本

- 每次评估需要 ~15-30 秒（编译+正确性验证+性能测试）
- LLM推理 30-80 秒/次
- 完整实验（48次评估）约 14-17 分钟
- GNAI速率限制可能中断长时间实验

### 7.6 搜索空间的局部性

- MAP-Elites的行为特征空间可能不够精细
- 所有1ms级kernel都落在同一细胞 (2,3,3,0)
- 无法区分v10的64WI架构和v11的128WI架构
- 可能导致优秀的非主流变体被淘汰

### 7.7 缺乏编译器感知

- LLM不了解Intel Graphics Compiler（IGC）的内部优化
- 某些代码模式可能阻碍编译器优化（如过多的SLM使用阻止自动寄存器分配）
- 没有feedback机制让LLM看到编译后的ISA

## 8. 方向性指导

### 8.1 近期工作（v12-v13）

**目标**: 在v10架构框架内做微调，从1.07ms优化到0.8-0.9ms

#### 策略A: 低温精细调优
```
种子: v10 best (1.07ms)
温度: 0.15-0.2
Instructions: 明确禁止架构变更, 仅允许微优化
方向:
  - 向量化SLM读取 (intel_sub_group_block_read_us)
  - 向量化B读取 (vload2, block reads)
  - 消除死代码 (K-remainder路径)
  - 尝试TILE_M=48或64 (保持B-global)
  - K循环展开2x
  - Prefetch B data
```

#### 策略B: 编译器引导优化
```
新增: 在评估流程中加入IGC ISA dump分析
让LLM看到实际编译产生的指令序列
识别编译器未能优化的热点（如不必要的spill/fill）
反馈到下一代的prompt中
```

### 8.2 中期工作

#### 自适应温度调度
```python
def adaptive_temperature(trial, best_score, prev_best_score):
    if trial < 3:
        return 0.3  # 初始高温探索
    improvement_rate = (best_score - prev_best_score) / prev_best_score
    if improvement_rate > 0.1:
        return 0.2  # 仍在快速改善, 中温
    elif improvement_rate > 0.01:
        return 0.15  # 渐进改善, 低温
    else:
        return 0.25  # 停滞, 升温寻找突破
```

#### 多阶段进化
```
阶段1 (T=0.3, 无种子): 探索基础架构空间
阶段2 (T=0.25, 阶段1最优为种子): 寻找架构突变
阶段3 (T=0.15, 阶段2最优为种子): 精细调优
```

#### 精细化行为特征
- 当前4D空间太粗（所有1ms kernel在同一格）
- 增加维度: WG_size_opt, SLM_usage_opt, tile_shape_opt
- 或改用连续坐标 + 自适应网格划分

### 8.3 长期方向

#### 1. 多级优化链路
```
LLM生成OCL → IGC编译 → ISA分析 → LLM理解ISA → 
→ 产生利于编译器优化的OCL模式
```

#### 2. 跨问题/跨硬件泛化
- 将v10的 "A-SLM + B-global" 策略迁移到其他GEMM尺寸
- 测试在不同Intel GPU（Arc A770, Data Center Max）上的效果
- 构建硬件-策略映射知识库

#### 3. 混合精度/算子融合
- 从纯GEMM扩展到 GEMM + Bias + ReLU 融合kernel
- FP16/INT8混合精度DPAS
- 支持不同的矩阵布局（row-major, column-major, blocked）

#### 4. 与传统autotuning互补
- 使用LLM进化发现架构原型
- 使用传统grid search精细调优数值参数（tile大小、展开因子）
- 组合两者优势

## 10. 结论

### 10.1 三个核心结论

1. **LLM进化式优化可以发现非直觉的高性能GPU kernel**: v10的"A-SLM + B-global"策略不来自教科书，而是进化搜索的"意外"产物。这验证了该方法在GPU优化领域的根本可行性。

2. **温度 × 种子 = 进化效果的决定因素**: 温度0.25配合"正确但非最优"的种子是最有效的组合。两个因素缺一不可——无种子的高温产生不了正确kernel，有种子的零温产生不了新架构。

3. **GPU微架构约束设置了当前方法的性能天花板**: ~21%的XMX利用率说明，纯代码层面的优化已接近极限。进一步突破需要编译器协同、ISA层面知识或硬件特性的更深层利用。

### 10.2 最终性能里程碑

```
参考基线:     33.9ms  (0.63 TFLOPS,  0.7% XMX)
v4 直接生成:  11.4ms  (1.88 TFLOPS,  2.0% XMX)
v10 进化最优: 1.07ms  (20.1 TFLOPS, 20.9% XMX)  ← 全局最佳
理论峰值:     0.22ms  (96.0 TFLOPS, 100% XMX)

总加速: 33.9ms → 1.07ms = 31.7x
进化贡献: 11.4ms → 1.07ms = 10.7x (在种子基础上的进化改善)
```
