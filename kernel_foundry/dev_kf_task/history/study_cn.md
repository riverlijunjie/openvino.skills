
# 目标问题

```
硬件: Intel Battlemage G21 (B580)
  - 20 Xe2 cores
  - FP16 XMX 理论峰值: 96 TFLOPS (20 × 2048 ops/cycle × 2.4 GHz / 1024)
  - 内存带宽: ~456 GB/s
  - L2 Cache: ~24 MB

问题: FP16 GEMM
  - C[2048, 2560] = A[2048, 2048] × B[2048, 2560]
  - 计算量: 21.5 GFLOP
  - 理论最小时间: 0.224 ms (计算密集型, OI=731 FLOP/byte)
  
关键指令: intel_sub_group_f16_f16_matrix_mad_k16(short8 a, int8 b, float8 acc)
  - 每次调用: 8×16×16 = 2048 FP16 MADs = 4096 FLOPs
  - Subgroup size = 16
```

# 实验总结

## 进化式优化的优势
 - 自动发现非直觉优化：很容易发现人工可能永远不会尝试的策略，从而获得意外的提升
 - 高效搜索：快速尝试不同的策略，效率远高于人类手工调优
 - 100%正确性保证: 进化框架的验证机制确保只有正确kernel进入精英档案,  消除了手工优化中"改快了但改错了"的风险
 - 知识积累效应：每一代的分析和代码都作为下一代的上下文，这种"思考→实验→反馈"循环模拟了人类工程师的学习过程


# 关键发现
- 温度是进化质量的核心控制变量：最优温度窗口: 0.25（足够产生架构突变，又不至于破坏已验证的结构）
- 种子Kernel是突破性能的关键前提: 种子的最佳选择是"正确但远未最优"的kernel
- 架构级相变现象是突破的契机：性能景观中存在适应度山谷——必须跨越一个"反直觉"的中间态才能到达全局最优
- GPU微架构约束要大于算法级优化: 在GPU优化中，占用率（occupancy）和延迟隐藏（latency hiding）的影响往往大于算法级优化（双缓冲、更大tile）
- LLM模型选择至关重要: 不同模型的代码风格差异导致"子代"无法在"父代"基础上有效进化, 较弱的模型（codex）可能破坏较强模型（opus）建立的优化结构

# 存在的问题与限制
- 对LLM推理能力的依赖: 较弱的模型很难完全进化，需要较强的模型
- 无法突破 ~21% 利用率天花板：进化到一定程度后无法继续突破
- 高温变异的不可控性: 最优温度高度依赖于当前种子质量,没有自动调温机制
- 种子选择的两难: 空种子：浪费进化预算在基础正确性上; 最优种子+高温：倾向于破坏已有优化
- 评估成本高:包括时间成本和token成本，每次实验40次迭代，基本3次实验就消耗完每天token的额度
- 搜索空间的局部性: MAP-Elites的行为特征空间可能不够精细, 可能导致优秀的非主流变体被淘汰
- 缺乏编译器感知：LLM似乎不了解Intel Graphics Compiler（IGC）的内部优化， 某些代码模式可能阻碍编译器优化

# 下一步方向
 - 低温精细调优
 - 编译器引导优化：在评估流程中加入IGC ISA dump分析， 让LLM看到实际编译产生的指令序列
 - 自适应温度调度: 初始高温探索, 中温快速改善, 低温渐进改善, 停滞升温寻找突破
 - 多阶段进化：探索基础架构空间，寻找架构突变，精细调优
 - 精细化行为特征: 当前4D空间太粗, 增加维度或改用连续坐标 + 自适应网格划分
