## 1. KernelFoundry 基本原理与演化逻辑

### 1.1 核心架构

KernelFoundry 采用 **MAP-Elites 质量多样性算法** 作为核心搜索框架，结合 LLM 作为语义级变异算子，
实现 GPU 内核代码的自动化演化优化。

```
┌─────────────────────────────────────────────────────────┐
│                   KernelFoundry 系统架构                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌───────────┐    变异请求    ┌───────────────┐          │
│  │ MAP-Elites │ ──────────── │  LLM 变异算子  │          │
│  │  行为空间   │ ◄─────────── │ (Claude Opus)  │          │
│  │  256 cells │   新内核代码   └───────────────┘          │
│  └─────┬─────┘                                          │
│        │ 选择父代                                         │
│        ▼                                                │
│  ┌───────────┐    编译+执行    ┌───────────────┐          │
│  │  岛屿模型   │ ──────────── │  硬件评估管线   │          │
│  │ 4 sub-pop  │ ◄─────────── │ Intel B580 GPU │          │
│  └───────────┘   适应度回传    └───────────────┘          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 行为空间定义

MAP-Elites 使用 **4维行为描述符**，每维 4 级 (0-3)，共 4^4 = 256 个行为格：

| 维度 | 含义 | 级别说明 |
|------|------|----------|
| `memory_opt` | 内存优化策略 | 0=无优化, 1=对齐, 2=SLM缓存, 3=2D Block IO |
| `compute_opt` | 计算优化策略 | 0=标量, 1=SIMD, 2=DPAS基础, 3=DPAS流水线 |
| `parallelism_opt` | 并行度策略 | 0=单SG, 1=多SG, 2=WG协作, 3=全芯片调度 |
| `esimd_opt` | ESIMD/扩展指令 | 0=无, 1=基础, 2=高级, 3=完全ESIMD |

### 1.3 适应度函数

```
combined_score = perf_score + 3 × I(correct ∧ speedup > 0) × runtime_improvement
```

其中：
- `perf_score`: 性能分数（基于执行时间的归一化值）
- `I(correct ∧ speedup > 0)`: 正确性且有加速的指示函数
- `runtime_improvement`: 相对于基线的运行时间改进比例

该设计强调 **正确性优先**：错误的内核即使运行极快也得分为零。同时对性能提升给予 3 倍加权奖励，
鼓励在正确的基础上追求更高性能。

### 1.4 LLM 作为语义变异算子

与传统遗传算法使用随机位翻转不同，KernelFoundry 利用 LLM 理解代码语义进行变异：

| 对比项 | 传统 GA | KernelFoundry |
|--------|---------|---------------|
| 变异方式 | 随机位翻转/交叉 | LLM 语义理解重写 |
| 搜索空间 | 所有可能的位串 | 语法正确的代码空间 |
| 约束遵守 | 需要修复算子 | 原生遵守语法约束 |
| 架构变化 | 几乎不可能 | 可实现架构级重构 |
| 局部调优 | 高效 | 依赖温度控制 |

### 1.5 岛屿模型与迁移

系统维护 4 个子种群（岛屿），每个岛屿独立演化，定期进行精英迁移：
- 每个岛屿有独立的 MAP-Elites 档案
- 迁移策略：每 N 代选择各岛屿 top-K 个体交换
- 目的：维持种群多样性，避免过早收敛

## 2. 实验条件与结果总结

### 2.1 目标问题

```
硬件: Intel Arc B580 (Battlemage)
  - 20 Xe2 cores
  - 96 TFLOPS FP16 XMX 峰值算力
  - ~456 GB/s 显存带宽
  - 32MB L2 缓存

问题: C[2048,2048] = A[2048,2560] × B[2560,2048]
  - 数据类型: FP16
  - 总运算量: 21.5 GFLOP
  - 理论最小时间: 0.224ms (达到峰值算力时)

关键指令: intel_sub_group_f16_f16_matrix_mad_k16(short8 a, int8 b, float8 acc)
  - 每次调用: 4096 FLOPs
  - 需要正确的数据布局和子组协作
```

### 2.4 关键架构对比

#### 旧架构 (v4-v14): SLM + 64 Work Items

```opencl
// 工作组: 64 WIs = 4 subgroups × 16 lanes
// 分块: 32×64×32 (M×N×K)
// 数据流: global → SLM → register (双缓冲 + barrier)

__attribute__((intel_reqd_sub_group_size(16)))
__kernel void matmul(...) {
    __local half A_slm[32][32];  // A 通过 SLM 中转
    // B: 从 global/L2 标量读取
    
    for (int k = 0; k < K; k += 32) {
        // 加载 A 到 SLM (barrier 同步)
        barrier(CLK_LOCAL_MEM_FENCE);
        // DPAS 计算
        acc = intel_sub_group_f16_f16_matrix_mad_k16(a, b, acc);
    }
}
```

**瓶颈**: SLM 带宽受限, B 矩阵读取效率低, WG 过小无法隐藏延迟

#### 新架构 (v15): 2D Block IO + 512 Work Items

```opencl
// 工作组: 512 WIs = 32 subgroups × 16 lanes
// SG布局: 4×8 grid, 每个SG处理: 32×32
// WG分块: 128M × 256N
// 数据流: global → register (2D Block IO 直达)

__attribute__((intel_reqd_sub_group_size(16)))
__kernel void matmul(...) {
    // A: 2D Block Read 直接从全局内存到寄存器
    short8 a = intel_sub_group_2d_block_read_16b_32r16x2c(...);
    // B: VNNI 变换读取
    int8 b = intel_sub_group_2d_block_read_transform_16b_16r16x2c(...);
    
    // 无 SLM, WG Swizzle (SWIZZLE=2) 实现 L3 复用
    // 4x K-unroll (128/iter) + 交错加载/计算
    acc = intel_sub_group_f16_f16_matrix_mad_k16(a, b, acc);
}
```

**优势**: 消除 SLM 瓶颈, 寄存器直通, 大 WG 隐藏延迟, L3 自动复用

#### 架构性能对比

| 指标 | 旧架构 (v12 最佳) | 新架构 (v15 最佳) | 提升倍数 |
|------|-------------------|-------------------|----------|
| 执行时间 | 0.948ms | 0.274ms | 3.46× |
| XMX 利用率 | 23.6% | 81.6% | 3.46× |

| 数据加载 | SLM中转 | 2D Block IO | - |

### 2.5 实验阶段分析

**阶段一: 冷启动探索 (v4-v9)**
- 目标: 从零开始找到可工作的 DPAS 内核
- 结果: v4 达到 11.4ms，后续 v5-v9 均无法超越
- 教训: 无种子的冷启动极度低效，大量预算浪费在正确性验证上

**阶段二: 热启动微调 (v10-v14)**
- 目标: 基于已有最佳结果进行渐进优化
- 结果: v10 实现相变跳跃(11.4→1.07ms)，v12 进一步推至 0.948ms
- 教训: 种子质量决定演化天花板，约束设计影响收敛效率

**阶段三: 架构跃迁 (v15)**
- 目标: 使用外部发现的新架构作为种子
- 结果: 直接达到 0.274ms (81.6% XMX)
- 教训: 架构选择 >> 微优化，正确的架构是一切的基础

## 3. 系统优劣势分析

### 3.1 核心优势

#### (1) 语义级变异能力

LLM 变异算子能够理解代码结构和 GPU 编程范式，产生语法正确、语义合理的变异体。
这是传统遗传算法完全无法做到的。

```
传统 GA:  random_bit_flip(kernel_code) → 99.9% 编译失败
KernelFoundry: LLM_mutate(kernel_code, context) → ~90% 编译成功
```

#### (2) 相变发现能力

v10 展示了系统发现相变（phase transition）的能力：
- Trial 6: 从 "A+B 均通过 SLM" 突变为 "A-SLM + B-global"
- 性能跳跃: 11.4ms → 1.07ms (10.7× 提升)
- 这种架构级重构是 LLM 变异的独特优势

#### (3) 质量多样性框架

MAP-Elites 天然维持解空间的多样性，避免所有个体收敛到同一局部最优：
- 256 个行为格各自保存精英
- 不同优化策略的组合被自动探索
- 即使当前最优退化，其他格中仍有备选方案

#### (4) 快速原型验证

完整的编译-执行-评估管线实现自动化闭环：
- 平均每个变异体评估时间 ~20-25s
- 包含正确性验证（数值比对）
- 无需人工干预即可运行数百次试验

#### (5) 约束引导的搜索空间剪枝

通过 DO NOT 约束有效缩小搜索空间：
- v12 的严格约束将正确率保持在 95.8%
- 同时实现了从 1.07ms 到 0.948ms 的改进
- 证明了约束设计对演化效率的关键作用

### 3.2 核心劣势

#### (1) 高退化率 (30-70%)

大量变异体性能显著退化，浪费计算预算：

```
v11 (T=0.3): ~40% 变异体退化到 >5ms (从 1.07ms 种子出发)
v13 (宽松):  ~50% 变异体退化
v15 (新架构): 双峰分布 — 0.274ms 或 1.2-13ms, 中间无过渡
```

#### (2) 多样性坍缩

在约束条件下 MAP-Elites 退化为单格主导：
- 大部分精英集中在少数行为格中
- 256 格中实际活跃格远少于理论值
- 与 MAP-Elites 的设计初衷相矛盾

#### (3) 编译器不透明性

LLM 无法观察 Intel IGC 编译器的行为：
- 无法预知寄存器分配结果
- 无法预判指令调度决策
- 产生 "源码层面正确但编译后低效" 的代码
- 编译器优化/反优化对最终性能影响巨大但不可见

#### (4) 信息过载导致正确率下降

| 实验 | 约束/提示数量 | 正确率 |
|------|---------------|--------|
| v12 | 少量严格 DO NOT | 95.8% |
| v13 | 无约束 | 87.5% |
| v14 | 37 条 Tips | 75.0% |

过多的优化建议使 LLM 过度修改代码，破坏正确性。

#### (5) 无自适应温度调度

系统使用固定温度，而最优温度取决于当前演化阶段：
- 探索阶段需要高温 (0.25-0.3)
- 微调阶段需要低温 (0.15-0.2)
- 当前需要人工在实验间手动调整

#### (6) 架构天花板问题

在给定架构内，演化很快收敛到局部最优：
- v10-v14 在旧架构上花费 ~300 个变异体
- 性能从 11.4ms 优化到 0.948ms (12×)
- 但距离理论极限仍有 4.2×
- 没有外部架构创新，系统无法突破天花板

#### (7) 模型依赖性

仅 Claude Opus 能产生有效变异：
- v5 (多模型混合): 完全失败
- Sonnet/Codex 对 GPU 内核优化理解不足
- 系统实质上高度依赖单一模型的能力

### 4.2 种子选择策略

种子质量是决定演化成功与否的最关键因素：

| 种子状态 | 预期效果 | 推荐策略 | 实验证据 |
|----------|----------|----------|----------|
| 无种子 | 极差 — 大量预算浪费在正确性 | 避免使用 | v5-v9 全部失败 |
| 正确但远离最优 | 良好 — 有大量改进空间 | T=0.25 自由探索 | v10: 11.4→1.07ms |
| 接近当前架构最优 | 有限 — 需要精细调优 | T=0.2 严格约束 | v12: 1.07→0.948ms |
| 外部强种子 (正确架构) | 最佳 — 快速收敛 | T=0.25 + 约束 | v15: 0.34→0.274ms |

```
          └────────┴────────┘
```

**有效约束示例 (v12 模式)**:
```
DO NOT:
- 移除 DPAS 指令
- 减少工作组大小
- 移除 SLM 双缓冲
- 将 K 循环步长减小到 32 以下
```

**无效约束示例 (v14 模式)**:
```
Tips 1-37:
- 尝试使用 prefetch 指令
- 考虑增大 K 循环展开因子
- 可以尝试不同的 SLM bank 布局
- 建议使用寄存器重排...
... (过多建议导致 LLM 过度修改)
```

### 4.5 预算分配建议

根据实验数据推荐的预算分配：

| 阶段 | 推荐迭代数 | 分支数 | 总变异体 | 预期墙钟 |
|------|-----------|--------|----------|----------|
| 初始验证 | 2-4 | 2 | 4-8 | 2-5 min |

**总预算**: 单次架构约 ~150-200 个变异体, 45-60 分钟

## 5. 提升演化效率与避免高概率退化

### 5.1 退化问题的根因分析

退化（regression）是 KernelFoundry 面临的核心挑战。分析显示退化有以下来源：

```
退化来源分布 (估计):

┌──────────────────────────┐
│ 编译器反优化     35%      │ ████████████
│ 数据布局破坏     25%      │ ████████
│ 同步/barrier错误  20%     │ ██████
│ 循环结构变化     12%      │ ████
│ 寄存器溢出       8%       │ ███
└──────────────────────────┘
```

### 5.2 七大改进建议

#### 建议一: 自适应温度调度

```python
# 伪代码: 基于改进率的温度调度
def adaptive_temperature(history, base_temp=0.25):
    recent_improvement = calc_improvement_rate(history[-5:])
    
    if recent_improvement > 0.1:   # 快速改进中
        return base_temp * 0.8     # 降温，精细调优
    elif recent_improvement < 0.01: # 停滞
        return base_temp * 1.2     # 升温，增加探索
    else:
        return base_temp           # 保持当前温度
```

**预期收益**: 减少 20-30% 的退化变异体，同时保持突破能力

#### 建议二: 渐进式约束生成

```
演化初期:  无约束 (允许自由探索)
     │
     ▼ 发现第一个退化模式
演化中期:  添加 DO NOT 规则 1
     │
     ▼ 发现第二个退化模式
演化后期:  添加 DO NOT 规则 1 + 2 + 3
```

基于观察到的失败模式动态生成约束，而非预设大量规则。
每当检测到某类退化反复出现，自动提取规则加入约束列表。

#### 建议三: 性能阈值过滤

```python
# 父代选择时加入性能门槛
def select_parent(archive, current_best):
    threshold = current_best * 1.5  # 仅保留 1.5× 以内的个体
    candidates = [ind for ind in archive if ind.time < threshold]
    return random.choice(candidates)
```

**原理**: 防止退化严重的个体作为父代产生更多退化后代。
当前 MAP-Elites 可能选择性能差但行为独特的个体作为父代。

#### 建议四: 分阶段架构搜索

将演化过程明确分为两个解耦的阶段：

```
阶段 A: 架构搜索 (人工 + LLM 辅助)
  - 目标: 找到正确的内存访问模式和并行策略
  - 方法: 参考硬件文档、vendor 示例代码、学术论文
  - 输出: 2-3 个候选架构的正确实现

阶段 B: 内核微调 (KernelFoundry 自动化)
  - 目标: 在确定架构内最大化性能
  - 方法: MAP-Elites + LLM 变异 + 约束引导
  - 输出: 该架构下的最优实现
```

v15 的成功证明了这种分离的有效性。

#### 建议五: 编译器反馈集成

```
当前流程:
  源码 → IGC编译 → 执行 → 时间
         (黑盒)

改进流程:
  源码 → IGC编译 → ISA分析 → 执行 → 时间
         │              │
         ▼              ▼
     编译警告       寄存器压力
     溢出信息       指令密度
                   管线利用率
```

将 IGC 编译器的中间产物（ISA dump、寄存器分配统计、溢出报告）
纳入 LLM 的上下文，使其能够理解 "为什么这段代码慢"。

#### 建议六: 精简精确约束优于丰富指导

实验明确证实：

```
DO NOT 列表 (3-5 条) >>> Tips 列表 (30+ 条)

原因:
1. DO NOT 约束空间小，LLM 容易遵守
2. Tips 诱导 LLM 同时尝试多种修改，增加出错概率
3. 少量精准约束保持代码稳定性骨架
4. 大量建议使 LLM 注意力分散
```

#### 建议七: 强外部种子 + 约束演化

v15 的成功模式应当成为标准流程：

```
1. 通过外部手段获取强种子:
   - 参考 vendor 优化示例
   - 移植其他平台的高性能实现
   - 人工设计新架构原型
   
2. 验证种子正确性

3. 使用 KernelFoundry 进行约束演化:
   - T=0.2-0.25
   - 严格 DO NOT 约束 (保护架构骨架)
   - 12-20 迭代
   
4. 分析结果, 提取改进模式, 反馈到种子设计
```

### 5.3 避免退化的具体策略

| 退化类型 | 检测方法 | 预防策略 |
|----------|----------|----------|
| 编译器反优化 | ISA 指令数暴增 | 约束循环结构不变 |
| 数据布局破坏 | 正确性失败 | DO NOT 修改数据类型/布局 |
| Barrier 错误 | 间歇性错误 | DO NOT 移动/删除 barrier |
| 寄存器溢出 | 性能突然恶化(>3×) | 限制活跃变量数量 |
| 架构降级 | 退回已知慢模式 | DO NOT 使用标量读取 |

### 5.4 监控指标与终止条件

在演化过程中应当监控以下指标来决定是否继续：

```python
# 建议的终止条件
STOP_CONDITIONS = {
    "no_improvement_trials": 8,      # 连续8次无改进则终止
    "correctness_drop_below": 0.80,  # 正确率低于80%则调整策略
    "regression_rate_above": 0.60,   # 退化率超过60%则降温
    "budget_exhausted": 80,          # 总变异体数超过80则终止
}
```

### 5.5 未来方向

1. **多目标演化**: 同时优化延迟、功耗、占用率
2. **迁移学习**: 将一个矩阵规模上的最优架构迁移到其他规模
3. **自动架构搜索**: 在架构空间中也使用演化搜索，而非仅限微调
4. **编译器协同设计**: 让 LLM 同时优化源码和编译器选项
5. **在线学习**: 从历史失败中学习，自动生成约束库

## 6. 总结

### 6.1 关键结论

1. **架构选择 >> 微优化**: v4-v14 在错误架构上花费约 300 个变异体仅获得 12× 改进；
   v15 在正确架构上 1 次试验即获得 3.46× 额外提升。

2. **KernelFoundry 最适合的场景**: 在已知正确架构上进行约束微调（v12, v15 模式），
   而非从零开始的架构探索（v5-v9 模式）。

3. **温度是最敏感的超参数**: 0.05 的温度差异可能导致完全不同的结果
   （v12: T=0.2 成功 vs v11: T=0.3 退化）。

4. **约束设计的核心法则**: "告诉模型不该做什么" 远比 "建议模型可以尝试什么" 有效。

5. **相变是可遇不可求的**: v10 的 10.7× 跳跃是偶然发现，系统无法可靠地触发此类事件。
