# v13 实验分析报告：宽松约束探索进化

## 1. 实验概述

| 参数 | 值 |
|------|------|
| 任务名 | b580_matmul_dpas_v13 |
| GPU | Intel B580 (Battlemage G21, 20 Xe2 cores) |
| 问题 | C[2048,2560] = A[2048,2048] × B[2048,2560], FP16, 21.5 GFLOP |
| 理论峰值 | 96 TFLOPS FP16 XMX, 理论最小时间 0.224ms |
| 模型 | claude-4-6-opus |
| 温度 | 0.25 |
| 迭代 | 20 trials × 4 branches = 80 kernel variants |
| 种子kernel | v12最佳 (0.948ms, 23.6% XMX, 双缓冲A+B-global, 64 WIs) |
| 总耗时 | 1758秒 (~29.3分钟) |
| 进化模式 | evolve_mode + gradient_tracking + optimization_aware_prompting |

## 2. 核心策略

v13采用**宽松约束探索**策略，与v12的严格约束形成对照实验：

### 2.1 USER_INSTRUCTIONS（宽松版）
```
- Current kernel achieves 0.948ms = 23% XMX utilization on B580 (96 TFLOPS peak).
    Hardware: B580 = 20 Xe2 cores, 96 TFLOPS FP16 XMX, 456 GB/s, 32MB L2, 128 KB SLM per core.
    DPAS: intel_sub_group_f16_f16_matrix_mad_k16(short8 a, int8 b, float8 acc)
- Focus on improving memory access efficiency and compute utilization.
- Adopts better thread walker and blocking strategy to maximize DPAS usage and hide memory latency.
- Provide explicit launch metadata (GWS/LWS/subgroup hints) in kernel comments.
```

**关键变化：移除了所有"DO NOT"约束，允许LLM自由探索架构变化。**

### 2.2 与v12配置对比
| 参数 | v12 | v13 |
|------|-----|-----|
| 温度 | 0.2 | 0.25 (更高探索性) |
| USER_INSTRUCTIONS | 3条"DO NOT"约束 | 无限制，仅给方向性指导 |
| 种子 | v10 best (1.07ms) | v12 best (0.948ms) |
| max_iters | 12 | 20 |
| gradient_sampling_weight | 0.3 | 0.2 |
| L2 cache说明 | 24MB | 32MB (修正) |
| SLM说明 | 未说明 | 128KB per core |

## 4. 进化动态分析

### 4.1 四阶段进化模式

**阶段1: 架构崩溃 (Trial 0-3)**
- 种子的双缓冲64-WI架构被LLM彻底重写
- 性能骤降到4.35-11.8ms
- LLM尝试了: 32×128(8SG), 32×256(16SG), 64×128(16SG), B-in-SLM
- 失败率高 (4/16 = 25%)
- 根因：宽松约束允许LLM改变已验证最优的架构

**阶段2: 部分恢复 (Trial 4-6)**
- LLM从失败中学习，找到32×256/16SG的可行架构
- 最佳1.14ms，但仍比种子慢20%
- Trial 5展现出稳定性（4个branch都在1.18-1.20ms）
- 使用 `intel_sub_group_block_read_us` 读取B（因为子组对B列对齐）

**阶段3: 持续震荡 (Trial 7-15)**
- 在1.16ms和3.07-5.15ms之间反复跳跃
- LLM不断尝试"更好"的架构（64×128 with A+B in SLM），反复失败
- 成功trial维持在32×256/16SG架构
- 失败trial是WG过大(>128 WIs)或B-in-SLM策略

**阶段4: 轻微收敛 (Trial 16-19)**
- Trial 16达到1.15ms（最终最佳）
- 但仍无法突破1.14ms的barrier
- Trial 19完全崩溃（6.05ms + 2次FAIL）

### 4.2 LLM探索的架构变体

| 架构 | WG大小 | Tile | SLM策略 | 最佳性能 | 评价 |
|------|--------|------|---------|---------|------|
| 32×64, 4SG (种子) | 64 | 32×64×32 | A双缓冲 | 0.948ms | 最优(v12证明) |
| **32×256, 16SG** | 256 | 32×256×16 | A双缓冲 | **1.14ms** | v13最佳 |
| 32×128, 8SG | 128 | 32×128×32 | A+B SLM | 3.09ms | B加载瓶颈 |
| 64×128, 16SG | 256 | 64×128×32 | A+B SLM | 4.35ms | 寄存器溢出 |
| 32×256, 16SG (K=64) | 256 | 32×256×64 | A SLM 40KB | 11.8ms | SLM过大 |

### 4.3 为什么32×256/16SG比种子差

种子(32×64/4SG)优于v13最佳(32×256/16SG)的原因：

1. **占用率**: 64 WIs在20个Xe2核上可同时运行更多WG；256 WIs限制了并发
2. **SLM效率**: 种子的4KB SLM留给更多WG共享核心；256WI版需要更多SLM
3. **B访问模式**: 种子每SG读16列B（连续），256版每SG也读16列但WG需要256列总带宽
4. **K tile大小**: 种子用K=32（2×k16），256版用K=16只做1×k16，barrier相对开销更大
5. **双缓冲效果**: 种子将A加载完全隐藏在计算后面；256版的A加载量更少但barrier更频繁

### 4.4 MAP-Elites状态

与v12相同，所有variant集中在cell `(2, 3, 3, 0)`：
- 最终精英score: 94.2 (vs v12的112.3)
- Island 0: 10 programs (主导)
- 其余Islands: 各1 program
- 有效多样性: 极低

## 5. 关键发现

### 5.1 约束的重要性被强力验证

| 指标 | v12 (严格约束) | v13 (宽松约束) |
|------|---------------|---------------|
| 最佳性能 | 0.948ms | 1.14ms (+20%) |
| Sub-1ms率 | 39.6% | 0% |
| 回归率(>2ms) | 33.3% | 72.9% |
| 失败率 | 4.2% | 12.5% |
| 探索效率 | 高 | 极低 |
| 种子超越 | ✅ 是 | ❌ 否 |

**结论：宽松约束不仅没有发现更好的架构，反而连种子的性能都无法复现。**

### 5.2 B580特定约束确认

v13的大量失败进一步确认了以下B580硬件约束：
- **B不应加入SLM**: 即使32×128/8SG配置，B-SLM的加载开销(64+ elements/WI)远超收益
- **WG不应超过64 WIs**: 256 WIs导致占用率下降
- **K=32 optimal**: K=16 barrier过多，K=64 SLM过大

### 5.3 LLM在宽松约束下的行为模式

1. **过度创新**: 有了"自由"后，LLM倾向于设计"理论上更优"但实际更差的架构
2. **无法回退**: 即使看到回归结果，LLM仍然在尝试变体而非回到种子架构
3. **分析正确但决策错误**: LLM正确地分析了瓶颈（B访问延迟），但选择了错误的解法（B-SLM）
4. **新架构学习缓慢**: 需要4-5轮失败才学会32×256/16SG是宽松约束下的可行点

## 7. 优化建议

### 7.1 v14方向（基于v12+v13教训）

1. **回到v12的严格约束策略**: 以0.948ms kernel为种子
2. **添加从v13学到的新"DO NOT"**:
   - DO NOT increase WG beyond 64 WIs
   - DO NOT use 32×256 tile (proven inferior to 32×64)
   - DO NOT put B in SLM
   - DO NOT use K-step smaller than 32
3. **新的允许探索方向**:
   - 尝试将双缓冲与K-loop展开2x结合
   - 更激进的B prefetch策略
   - 探索不同的SLM stride以避免bank conflict
   - 尝试异步拷贝(intel_sub_group_block_read for A loads)
4. **温度**: 保持0.2或降到0.15

### 7.2 框架层面改进

1. **种子保护机制**: 保证每trial至少1个branch直接使用种子的exact结构
2. **performance gate**: 自动丢弃比种子慢2x以上的variants，不让其进入精英库
3. **架构锁**: 允许配置"锁定"某些代码结构参数(WG size, tile dims)

### 7.3 实验设计教训

- **约束不是限制，而是加速器**: 好的约束将搜索空间聚焦到有希望的区域
- **"自由创新"对LLM无效**: 在高度特化的硬件(B580 XMX)上，LLM的"理论直觉"不可靠
- **empirical > theoretical**: 编译器行为、硬件调度、occupancy都不可从代码表面推理

## 8. 结论

v13实验是一个**有价值的负面结果**：

1. 明确证明了**宽松约束劣于严格约束**
2. LLM在自由探索中发现了32×256/16SG架构(1.14ms)，但这仍比v12的种子差20%
3. 80个variants中0个达到种子的0.948ms水平，而v12有19个超越种子
4. 验证了v12报告中的所有"DO NOT"约束都是正确的
5. 额外确认: K=16 tile劣于K=32, 256 WIs劣于64 WIs

**核心结论**: 对于kernel micro-optimization, "约束驱动进化" >> "自由探索进化"。下一步应该在v12的严格约束基础上添加更精细的micro-optimization指导。
