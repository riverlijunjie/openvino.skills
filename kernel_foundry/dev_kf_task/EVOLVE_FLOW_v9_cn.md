## 1. 实验配置与目标

### 1.1 实验目标

v9 实验的核心假设：**在 v8 验证了显式 DPAS 类型约束 + 进化模式有效性的基础上，适度提高温度（0.1→0.2）能否同时获得更好的最终结果和更快的收敛速度？**

v8 已证明 temp=0.1 是进化模式的最低有效温度。v9 将温度提升一倍至 0.2，目标是在保持类型正确性的同时获得更大的架构多样性，从而让进化算法探索到更优的解空间区域。

### 1.4 实验设计要点

v9 与 v8 的**唯一区别是温度从 0.1 提升到 0.2**。这构成了一个严格的控制实验：

- 相同模型、相同提示、相同约束、相同评估预算
- 温度是唯一自变量
- 最佳运行时间和收敛速度是因变量

这一设计使得温度效应可以被精确隔离和量化。

### 3.4 停滞原因分析

Trials 3-7 的 20 次评估为何无法超越 23.8ms？

1. **架构锁定**：进化后期所有 branch 收敛到 ~39.8ms 的 "256 WIs / 16 SG" 架构
2. **探索陷阱**：梯度追踪记录的改进历史指向 "更大 tile + 更多 SG" 方向，但真正的最优解在 "512 WIs / 32 SG" 方向
3. **变异幅度不足**：temp=0.2 的变异强度不足以从 16-SG 架构跳跃到 32-SG 架构
4. **Elite 遮蔽效应**：23.8ms 的高分 (9.27) 使得稍差的变体无法进入种群，限制了多样性

### 3.5 Trial 4 的"近似突破"

Trial 4 Branch 2 达到 25.4ms（仅比最佳 23.8ms 慢 6.7%），但未触发 elite 替换。这说明进化算法确实在 23.8ms 附近的区域搜索，但缺乏足够的随机性来突破当前最优。

## 5. Kernel架构演进分析

### 5.1 v9 最佳 Kernel 架构（Trial 2, 23.8ms）

**工作组配置**：
```c
__attribute__((reqd_work_group_size(16, 32, 1)))
// 16 lanes × 32 subgroups = 512 work-items per workgroup
```

**Tile 分解**：
- 工作组处理 **64M × 64N** 的输出块
- 32 个子组排列为 **8(M) × 4(N)**
- 每个子组计算 **8×16** 的输出子块
- TILE_K = 32，每个 K-tile 调用 **2 次 DPAS (k16 each)**

**SLM 布局（含 padding）**：
```c
// A tile: 64 rows × (32+2) columns, padding 消除 bank conflict
#define A_STRIDE (TILE_K + 2)  // = 34
__local half A_slm[TILE_M * A_STRIDE];  // 64 × 34

// B tile: 32 rows × (64+2) columns
#define B_STRIDE (TILE_N + 2)  // = 66
__local half B_slm[TILE_K * B_STRIDE];  // 32 × 66
```

**SLM 加载策略**：
```c
// 512 WIs 协作加载 A tile: 64×32 = 2048 elements
// 每个 WI 加载 2048/512 = 4 个 half 元素
// 使用标量加载 + 边界检查
int load_idx = local_id;  // 0..511
int a_row = load_idx / TILE_K;   // 0..63
int a_col = load_idx % TILE_K;   // 0..31
if (global_m + a_row < M && k_base + a_col < K)
    A_slm[a_row * A_STRIDE + a_col] = A[...];
```

**DPAS 调用**：
```c
// 正确的操作数类型
short8 a_val = *((__local short8*)(A_slm + sg_m_start * A_STRIDE + k_offset));
int8   b_val = *((__local int8*)(B_slm_packed + ...));
float8 acc = (float8)(0.0f);

// 两次 k16 DPAS 调用覆盖 TILE_K=32
acc = intel_sub_group_f16_f16_matrix_mad_k16(a_val_0, b_val_0, acc);
acc = intel_sub_group_f16_f16_matrix_mad_k16(a_val_1, b_val_1, acc);
```

**关键性能特征**：
1. **512 WIs 高并行加载**：比 256 WIs 方案快 ~2x 的 SLM 填充速度
2. **SLM padding**：A stride=34, B stride=66，有效避免 bank conflict
3. **标量 SLM 加载 + bounds checking**：安全但每 WI 仅 4 元素，总体效率高
4. **每 subgroup 最小输出（8×16）**：降低寄存器压力，但牺牲了计算密度

### 5.2 39.8ms 收敛架构（Trials 3-7 主流）

**工作组配置**：
```c
__attribute__((reqd_work_group_size(16, 16, 1)))
// 16 lanes × 16 subgroups = 256 work-items per workgroup
```

**Tile 分解**：
- 工作组处理 64M × 64N
- 16 个子组排列为 4(M) × 4(N)
- 每个子组计算 **16×16** 的输出子块（两行 8×16 DPAS 块）

**为何 16-SG/256WI 架构比 32-SG/512WI 架构慢？**

| 对比维度 | 32 SG (23.8ms) | 16 SG (39.8ms) | 影响 |
|----------|----------------|----------------|------|
| WI 总数 | 512 | 256 | SLM 加载 2x 快 |
| 每 WI SLM 加载量 | 4 elements | 8 elements | 前者更轻 |
| 每 SG 输出大小 | 8×16 | 16×16 | 后者寄存器压力更大 |
| 计算/加载比 | 低 | 高 | 受限于加载带宽时前者更优 |
| SLM 总使用量 | 同等 | 同等 | 无差异 |

**关键结论：SLM 加载带宽是当前瓶颈**，而非计算单元利用率。更多 WIs 意味着更快完成协作 SLM 加载，即使每个子组的计算量更小。

### 5.3 Trial 5 回退分析（90.8ms, 91.0ms）

Trial 5 的两个 branch 尝试了 **TILE_K=64** 策略：

```c
// 每次 SLM 加载 64 列（翻倍）
// A tile: 64 × 64 = 4096 elements → 每 WI 加载 16 elements
// B tile: 64 × 64 = 4096 elements → 每 WI 加载 16 elements
```

**失败原因**：
1. **SLM 容量压力**：单次加载 8KB+ 的 SLM 数据，接近 B580 的 64KB SLM 限制
2. **加载延迟增加**：每 WI 需加载 16 个元素（vs 标准 4 个），barrier 前延迟翻 4 倍
3. **K-loop 迭代减少**的收益远不及加载延迟增加的代价

这是温度 0.2 产生的"高方差"效应的典型表现——模型尝试了激进优化（TILE_K 翻倍），但效果适得其反。

### 5.4 架构演进总图

```
Trial 0: 32×32 tile, 8 SG, 128 WI → 53.4ms (基础架构)
    ↓ 扩大 tile + 增加 SG
Trial 1: 64×64 tile, ? SG, ? WI → 32.3ms (过渡态)
    ↓ 最大化 WI 数量
Trial 2: 64×64 tile, 32 SG, 512 WI → 23.8ms ★ (SLM带宽最优化)
    ↓ 进化停滞，收敛到次优
Trials 3-7: 64×64 tile, 16 SG, 256 WI → 39.8ms (局部最优陷阱)
```

**悖论**：进化算法在找到全局最优（23.8ms）后，后续迭代反而收敛到了次优解（39.8ms）。这是因为梯度追踪系统记录的改进方向是"增加每 SG 计算量"，而真正的最优方向是"增加总 WI 数量"。

## 6. 关键技术发现

### 6.1 发现一：温度 0.2 优于 0.1，且优势显著

**定量证据**：

| 对比维度 | temp=0.1 (v8) | temp=0.2 (v9) | 优势方 |
|----------|---------------|---------------|--------|
| 最佳运行时间 | 30.9ms | 23.8ms | v9 (+23%) |
| 收敛速度 | Trial 7 | Trial 2 | v9 (5x 快) |
| 正确率 | 68.75% | 93.75% | v9 (+36%) |
| 加速比 | 1.10x | 1.42x | v9 (+29%) |

**所有核心指标 v9 均优于 v8**。温度 0.2 在进化模式下是严格优势策略。

**机制解释**：
- temp=0.1 的变异幅度仅够在"同一架构族"内微调参数（如 tile 大小从 32 变为 64）
- temp=0.2 的变异幅度足以跨越"架构族边界"（如从 8-SG 跳到 32-SG 方案）
- 更大的架构多样性使得进化算法更快发现性能峰值

### 6.2 发现二：进化前 3 轮是关键窗口期

**数据支撑**：
- Trial 0-2：4 次 elite 替换，运行时间从 ∞ 降至 23.8ms
- Trial 3-7：**0 次** elite 替换，20 次评估全部浪费

这意味着 v9 的 32 次评估中，仅前 10-12 次有意义，后续 20 次评估的边际价值为零。

**操作含义**：应在进化停滞检测后**动态终止或切换策略**，而非机械地执行全部迭代。

### 6.3 发现三：高 WI 数量 > 高寄存器利用率

**经验规律**：
```
512 WI / 32 SG / 每SG算 8×16  → 23.8ms ★
256 WI / 16 SG / 每SG算 16×16 → 39.8ms
128 WI /  8 SG / 每SG算 8×16  → 53.4ms
```

在 Intel Battlemage G21 上的 DPAS matmul 中：
- **SLM 协作加载是主要瓶颈**（非计算单元）
- 更多 WIs = 更快填充 SLM = 更高吞吐
- 每 SG 的计算量增加（更多寄存器分块）在当前负载下不构成瓶颈

**硬件推测**：B580 的 DPAS 单元延迟较高但吞吐有限，64×64 tile 的计算量不足以隐藏 SLM 加载延迟，因此加载速度（取决于 WI 数）成为决定性因素。

### 6.4 发现四：显式类型约束在高温下依然有效

| 温度 | 有类型模板 | 类型正确率 | 实验 |
|------|-----------|-----------|------|
| 0.0 | 无 | 0% | v7 |
| 0.1 | 有 | **100%** | v8 |
| 0.2 | 有 | **100%** | v9 |
| 0.3-0.6 | 无 | 0% | v5 |

**结论**：USER_INSTRUCTIONS 中的显式 DPAS 类型模板具有极强的约束力，能在 temp=0.0 到 temp=0.2 的范围内可靠生效。这是 v8/v9 相对 v5/v7 成功的基础保证。

### 6.5 发现五：温度增加带来方差增加但上界也提升

```
温度 0.1 (v8): 最佳 30.9ms, 最差 332ms, 方差大
温度 0.2 (v9): 最佳 23.8ms, 最差 124ms, 方差也大但上下界均更优

例外：Trial 5 的 90ms 回退（TILE_K=64 激进优化失败）
```

**结论**：温度 0.2 的方差特征是"高回报的赌注"——偶尔产生失败（90ms），但更频繁地产生突破（23.8ms）。在有 4 branches/trial 的设置下，只需 1 个 branch 成功即可，这使得方差成为优势而非劣势。

### 6.6 发现六：进化模式仍有 2.1x 差距需要弥补

| 方案 | 最佳运行时间 | 相对理论最优 |
|------|-------------|-------------|
| v4（直接生成） | 11.4ms | 基准 |
| v9（进化模式最佳） | 23.8ms | 2.09x 慢 |
| 理论目标 | ~5ms | 0.44x |

进化模式虽然在 v9 取得了重大进步，但仍远逊于 v4 的直接生成。这强烈暗示：
1. 进化上下文对模型输出质量有负面影响
2. 或者进化搜索空间的初始化方向错误（从头开始而非从 v4 最优解开始）

## 总结

v9 通过将温度从 0.1 提升至 0.2 这一**唯一变更**，在所有核心指标上全面超越 v8：

- **性能**：23.8ms vs 30.9ms（提升 23%）
- **收敛**：Trial 2 vs Trial 7（快 5 轮）
- **正确率**：93.75% vs 68.75%（提升 36%）
- **加速比**：1.42x vs 1.10x

这证明了温度是进化模式中最关键的超参数之一。0.2 在架构多样性和代码正确性之间取得了优秀的平衡。

然而，v9 同样暴露了两个核心瓶颈：

1. **早期停滞问题**：62.5% 的评估预算被浪费在无法突破 23.8ms 的搜索上
2. **与 v4 的差距**：进化模式最佳（23.8ms）仍是直接生成最佳（11.4ms）的 2.1 倍

下一步的核心方向是：**从 v4 的 11.4ms kernel 温启动 + 温度 0.25 + 早停机制**，目标突破 11.4ms 的历史最佳记录。
