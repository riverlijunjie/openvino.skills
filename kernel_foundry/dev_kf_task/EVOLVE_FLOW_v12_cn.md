# v12 实验分析报告：约束微调进化

## 1. 实验概述

| 参数 | 值 |
|------|------|
| 任务名 | b580_matmul_dpas_v12 |
| GPU | Intel B580 (Battlemage G21, 20 Xe2 cores) |
| 问题 | C[2048,2560] = A[2048,2048] × B[2048,2560], FP16, 21.5 GFLOP |
| 理论峰值 | 96 TFLOPS FP16 XMX, 理论最小时间 0.224ms |
| 模型 | claude-4-6-opus |
| 温度 | 0.2 |
| 迭代 | 12 trials × 4 branches = 48 kernel variants |
| 种子kernel | v10最佳 (1.07ms, 20.9% XMX利用率) |
| 总耗时 | 1105秒 (~18.4分钟) |
| 进化模式 | evolve_mode + gradient_tracking + optimization_aware_prompting |

## 2. 核心策略

v12是首次采用**约束微调**策略的实验：

### 2.1 USER_INSTRUCTIONS 约束
```
DO NOT change the fundamental architecture (this was proven best).
DO NOT add B to SLM (causes regression).
DO NOT increase WG beyond 64 WIs (causes regression).
```

### 2.2 允许的微优化方向
1. `intel_sub_group_block_read_us` 替代标量SLM读取
2. 合并B的标量读取为 vload2 或 block reads
3. 移除K余数路径 (K=2048可被32整除)
4. 尝试 TILE_M=48 或 64
5. K循环展开2x
6. 添加prefetch预取下一个B tile

### 2.3 关键配置差异 (vs v11)
| 参数 | v11 | v12 |
|------|-----|-----|
| 温度 | 0.3 | 0.2 (更保守) |
| USER_INSTRUCTIONS | 无约束 | 严格约束架构 |
| 种子 | v10 best (1.07ms) | 相同 |
| 模型 | claude-4-6-opus | 相同 |
| max_tokens | 10000 | 10000 |

## 4. 进化动态分析

### 4.1 三阶段进化模式

**阶段1: 快速收敛 (Trial 0-5)**
- 从种子1.07ms快速优化到0.959ms
- 核心突破：`intel_sub_group_block_read_us` + 移除K余数路径 + 双缓冲
- 每轮都有显著改进

**阶段2: 探索陷阱 (Trial 6-8)**
- 性能骤降到1.71-2.21ms
- 原因：LLM尝试了过度优化（如B block reads、全局边界消除）
- 这些"理论上更好"的技术实际导致编译器生成更差的代码
- Trial 8出现了一次correctness failure

**阶段3: 恢复与突破 (Trial 9-11)**
- Trial 9恢复到0.948ms (新全局最优)
- MAP-Elites archive保留了精英，使LLM能选择好的父代重新进化
- 但Trial 10-11又出现回归

### 4.2 MAP-Elites行为空间

由于架构约束，所有variant落入同一个cell `(2, 3, 3, 0)`:
- 行为多样性极低（仅1/256 cells覆盖）
- 精英替换频繁：score从58.2→112.3（8次替换）
- 岛模型高度不平衡：Island 0有10个程序，其他各1个

这表明约束微调模式下MAP-Elites的多样性机制基本失效，实验退化为纯粹的精英选择+变异。

### 4.3 优化技术效果

| 技术 | 效果 | 典型运行时间 |
|------|------|-------------|
| `intel_sub_group_block_read_us` (SLM A读取) | ✅ 有效 | 减少~50μs |
| 双缓冲 SLM A | ✅ 有效 (最佳方案) | 0.948-0.970ms |
| K循环展开2x | ✅ 有效 | 0.968-0.970ms |
| 移除K余数路径 | ✅ 有效 | 减少代码压力 |
| B block reads (全局) | ❌ 回归 | ~2.1ms |
| 全局边界检查消除 | ❌ 回归 | ~1.7-2.1ms |
| `vload2` for B | ⚠️ 中性 | 无显著改善 |
| SLM stride = 32 (无padding) | ✅ 有效 | vs stride=34 |

## 5. 最佳Kernel分析 (0.948ms)

### 5.1 架构

```
WG: 64 WIs (4 subgroups × 16 lanes)
Tile: 32×64×32 (M×N×K per iteration)
SLM: 双缓冲, 2×(32×32×2B) = 4KB total
A: SLM (双缓冲, block_read_us读取)
B: Global/L2 (标量读取, col-clamped)
DPAS: intel_sub_group_f16_f16_matrix_mad_k16
K-loop: 64 tiles, 无余数路径
```

### 5.2 关键优化点

1. **双缓冲A载入**: 加载下一个A tile与当前tile的DPAS计算重叠
2. **`intel_sub_group_block_read_us`**: 向量化SLM读取，替代标量`as_short(slm[...])`
3. **列夹紧**: `b_col = col_valid ? col_idx : (N-1)` 消除内循环分支
4. **预计算偏移**: A载入的行/列偏移在循环外计算一次
5. **ushort类型一致性**: 避免half↔short类型转换开销

### 5.3 性能分解

```
理论计算时间: 0.224ms (96 TFLOPS)
实际时间:     0.948ms
利用率:       23.6%
瓶颈:         B矩阵全局内存访问延迟
  - 每k16步需要16次标量读取 (8个ushort对)
  - 总B读取: 64 tiles × 2 k16步 × 16 × 4 SGs = 8192次全局读取
  - L2命中率高(24MB L2可容纳B整体~10MB)，但延迟仍是瓶颈
```

## 7. 问题与发现

### 7.1 回归问题 (Trial 6-8, 11)
- 33%的variants性能大幅回归到2.0ms+
- 根因：LLM生成的代码虽然"逻辑正确"，但编译器无法有效优化
- 典型失败模式：B block read尝试、过度内联、复杂控制流

### 7.2 多样性坍塌
- 256-cell空间仅覆盖1个cell
- 约束微调使所有variant行为特征相同
- MAP-Elites退化为 (1+λ)-ES 策略

### 7.3 进化不稳定性
- Trial 4-5达到0.959ms后，Trial 6-8骤降到2.1ms
- 说明进化过程中"好的变异"和"坏的变异"比例约为2:1
- 低温(0.2)并不能完全避免有害探索

## 8. 优化建议

### 8.1 短期（v13方向）
1. **更严格的约束**: 在USER_INSTRUCTIONS中明确禁止B block reads和全局边界消除
2. **降温至0.1**: 进一步减少有害变异
3. **种子更新**: 使用0.948ms双缓冲kernel作为新种子
4. **聚焦B访问优化**: 尝试B的预取(intel_sub_group_block_prefetch)和内存布局重排

### 8.2 中期（架构层面）
1. **精英保护**: 禁止从回归variant进化
2. **性能阈值过滤**: 只保留 <1.2ms 的variants作为父代
3. **自适应约束**: 根据前N轮结果动态添加/移除约束

### 8.3 长期（框架改进）
1. **编译器反馈集成**: 将IGC编译器的寄存器分配、调度信息反馈给LLM
2. **指令级分析**: 使用IGC disassembler分析实际生成指令
3. **多目标优化**: 同时优化延迟和功耗/面积效率

## 9. 结论

v12实验证明了**约束微调**策略的有效性：

1. **USER_INSTRUCTIONS约束** 成功防止了v11中出现的架构级回归
2. **低温(0.2)** 保证了大部分变异在合理范围内
3. **双缓冲** 是超越1ms barrier的关键技术
4. 最终达到 **0.948ms / 23.6% XMX利用率**，创下新的最佳纪录
5. 但仍有33%的variants回归，说明约束需要进一步细化

v12的核心教训：**"告诉LLM不要做什么"比"告诉LLM做什么"更有效**。明确的负面约束（DO NOT）比正面建议（Try...）产生了更一致的高质量输出。
