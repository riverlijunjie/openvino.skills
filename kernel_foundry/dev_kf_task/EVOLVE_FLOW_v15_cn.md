## 2. 核心策略

v15采用**全新架构种子**策略，放弃了v4-v14的SLM+64WI架构，转向完全基于2D Block IO的大WG设计。

### 2.1 种子Kernel架构 (v28 FINAL)

```
- WG: 512 WIs (32 subgroups × 16 lanes)
- SG布局: 4×8 grid (WG_M=4, WG_N=8)
- 每SG: 32×32 output tile
- WG Tile: 128M × 256N
- K-step: 32 per iteration
- A加载: intel_sub_group_2d_block_read_16b_32r16x2c (32行×32列)
- B加载: intel_sub_group_2d_block_read_transform_16b_16r16x2c (VNNI变换)
- C存储: intel_sub_group_2d_block_write_16b_8r16x1c
- SLM: 不使用（纯寄存器+2D block IO）
- WG Swizzle: SWIZZLE=2 (L3 B-panel复用)
- DPAS: 8次 intel_sub_group_f16_f16_matrix_mad_k16 per K-step per SG
```

### 2.2 USER_INSTRUCTIONS

仍保留v14的完整37条oneDNN Tips + 旧架构的DO NOT约束。

**注意**: USER_INSTRUCTIONS中描述的"64 WIs, SLM, TILE 32x64x32"与新种子架构完全不匹配，形成一种"松散约束"环境——LLM必须自行理解新架构的正确优化方向。

### 4.3 为什么0.274ms是当前极限

分析0.274ms的性能构成:
```
理论计算时间 = 21.5 GFLOP / 96 TFLOPS = 0.224ms
实际时间 = 0.274ms
效率 = 0.224 / 0.274 = 81.6%
额外开销 = 0.274 - 0.224 = 0.050ms (18.4%)
```

开销来源分析:
1. **内存带宽约束** (~30%): A矩阵2048×2560×2B = 10.5MB需全部读取，B部分复用
2. **WG启动/结束开销** (~5%): 128 WGs的dispatch和drain
3. **L2/L3 miss** (~10%): 首次访问B数据需从HBM加载
4. **DPAS pipeline bubble** (~5%): K-loop迭代间的pipeline gap

## 5. 关键发现

### 5.1 架构跃迁的颠覆性效果

| 指标 | 旧架构 (v12最佳) | 新架构 (v15最佳) | 提升幅度 |
|------|-----------------|-----------------|---------|
| 运行时间 | 0.948ms | **0.274ms** | **3.46×** |
| XMX利用率 | 23.6% | **81.6%** | **3.46×** |
| 达成TFLOPS | 22.6 | **78.4** | **3.46×** |
| 距理论峰值 | 4.23× | **1.22×** | — |

**根因分析**: 旧架构(64 WIs, SLM)的根本瓶颈:
1. **WG太小(64 WIs)**: 无法饱和B580的执行资源
2. **SLM转存开销**: A从global→SLM→register的双重拷贝
3. **B标量读取**: 每次2个scalar read × 8次 vs 2D block read一次加载32×32
4. **缺乏L3复用**: 无WG swizzle，B的L3 miss率高
5. **K-step太小(32)**: 循环开销占比高

### 5.2 2D Block IO为什么优势巨大

| 特性 | SLM方案 (v12) | 2D Block IO (v15) |
|------|--------------|-------------------|
| A加载 | global→SLM→register (2次拷贝+barrier) | global→register直接加载 |
| B加载 | 标量对读 ×16次/K-step | 单次2D block read+VNNI |
| VNNI变换 | 手动重排 | 硬件自动(transform模式) |
| WG大小 | 64 WIs (4 SGs) | 512 WIs (32 SGs) |
| 指令数/K-step | ~100+ | ~20 |
| 延迟隐藏 | 依赖SLM double-buffer | 2D block + 4x K-unroll |
| L3复用 | 无 | WG swizzle |

### 5.3 oneDNN Tips在v15中的适用性

| Tips类别 | v14中的效果 | v15中的效果 |
|----------|------------|------------|
| 2D Block Load | ❌ 未尝试 | ✅ **种子核心**! |
| K-loop unroll | ❌ 有害 | ✅ **4x unroll关键优化** |
| Load/compute interleaving | ⚠️ 偶尔改善 | ✅ **0.417→0.274ms** |
| WG swizzle | ❌ 不适用 | ✅ **L3 B-panel复用** |
| SLM padding | ✅ 有效 | N/A (不用SLM) |
| Boustrophedon | ✅ 微小改善 | ⚠️ 无可测量差异 |
| Software pipeline | — | ❌ GRF溢出 |
| Prefetch | — | ❌ HW prefetch已足够 |
| 更大/更小WG | — | ❌ 32 SGs最优 |

**结论**: 在正确的架构基础上，oneDNN Tips中的K-unroll和interleaving真正发挥了效力。问题不在Tips本身，而在于v14的旧架构无法利用这些高级优化。

## 7. 优化建议

### 7.1 v16方向

v15已达到81.6% XMX利用率，距理论峰值仅18.4%。进一步优化空间:

1. **精准约束v15最佳架构**:
   ```
   DO NOT change from 32 SGs (4×8) layout
   DO NOT change from 4x K-unroll (128 per iteration)
   DO NOT add SLM (2D block IO is proven optimal)
   DO NOT add explicit prefetch (HW prefetch sufficient)
   DO NOT use software pipelining (causes register spill)
   DO NOT change per-SG tile from 32×32
   ```

2. **仅允许微调**:
   - 尝试不同的load/compute interleaving顺序
   - 微调SWIZZLE值(2 vs 4 vs 8)
   - 尝试在K-loop最后一次迭代中提前启动C写回
   - 尝试不同的A load分块(64行 vs 32行)
   - 调整GRF分配hints

3. **降低温度到0.15**: 在0.274ms附近精细搜索

4. **增加分支数到6**: 更多并行探索微小变体

### 7.2 突破81.6%的可能方向

要突破当前81.6%的上限，需要:

1. **Cooperative prefetch**: 让部分SG专门负责预取下一个WG tile的数据到L2
2. **Cross-iteration pipelining**: 不是software pipeline，而是让编译器看到更好的调度机会
3. **Named barriers**: 减少SG间同步开销(如果存在)
4. **更大WG tile + Stream-K**: 128×512的WG tile配合K-split
5. **双精度A读取**: 一次读64行而非32行A(如果GRF允许)

### 7.3 最佳实践总结更新

| 策略 | 效果 | 原因 |
|------|------|------|
| **正确的架构选择** | ✅✅✅ 决定性 | 2D Block IO vs SLM = 3.46× |
| 严格DO NOT约束 | ✅ 有效 | 防止LLM改变已验证最优 |
| K-unroll + interleave | ✅ 关键 | 隐藏内存延迟 |
| 低温度+微调 | ✅ 有效 | 聚焦搜索空间 |
| 大量Tips | ⚠️ 双刃剑 | 正确架构下有些有用，错误架构下有害 |
| 过度探索WG形状 | ❌ 浪费 | 32 SGs已验证最优 |

**核心教训**: 
- **"Architecture First" — 架构选择的影响远超微优化的累积效果**
- v4-v14花费6轮实验(~300个kernel variants)在错误架构上从11.4ms优化到0.948ms(12×提升)
- v15用1个trial就在正确架构上从~0.34ms达到0.274ms，且绝对性能是v12最佳的3.46×

## 8. 结论

v15实验是整个系列的**决定性突破**:

1. **革命性提升**: 0.274ms vs v12的0.948ms，**3.46倍性能飞跃**
2. **高效进化**: 仅1个trial（4个branch）就找到全局最优解
3. **高正确率**: 93.8%，仅次于v12(95.8%)
4. **81.6% XMX利用率**: 距理论极限(0.224ms)仅22%
5. **架构选择是王道**: 2D Block IO + 大WG + K-unroll + interleave的组合彻底碾压SLM方案

**关键发现**:
- 2D Block IO消除了SLM方案的所有瓶颈(双拷贝、barrier、标量读取)
- 4x K-unroll + interleaved load/compute是0.417→0.274ms的关键
- 32 SGs (4×8) 是B580上此问题的最优WG配置
- Hardware prefetch已足够，显式prefetch反而有害
- oneDNN Tips中的高级策略(K-unroll, interleave, 2D block, WG swizzle)在正确架构上全部发挥了作用

**v4-v15进化总结**:
```
v4  (冷启动):     11.4ms  — 2% XMX
v10 (温启动):     1.07ms  — 21% XMX  (10.7× over v4)
v12 (精调SLM):    0.948ms — 24% XMX  (旧架构极限)
v15 (新架构):     0.274ms — 82% XMX  (3.46× over v12, 41.6× over v4)
```

从11.4ms到0.274ms，总提升**41.6倍**。其中架构跃迁(SLM→2D Block IO)贡献了3.46×，占整体优化的主导地位。

```

- C存储: intel_sub_group_2d_block_write_16b_8r16x1c (8次)
- SLM: 不使用
```
